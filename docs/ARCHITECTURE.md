# agent.branch ‚Äî Architecture

Single source of truth for **what we build and why** ‚Äî product, system, services, data, **and** app/screen layout. All of it lives here because it's one *kind* of knowledge (decisions that translate into code), regardless of layer.

**Companion docs:**

- **`DESIGN.md`** ‚Äî the *visual design system* (theme, color tokens, type scale, spacing, components).

---

## 1. Product thesis

**agent.branch is a skill *testing / CI* tool that also authors.** That is the long-term thesis, and it's the sharp end of the pitch ‚Äî most tools stop at editing; the durable value is *validating* a skill before it ships.

In the near term it's a **visual authoring + lightweight-validation tool for agent skills**: build a skill in chat with a live-streaming preview, **visualise** its logic, **test-run** it, check its **triggering**, and **export** it. The frame is **Claude-first runtime, standard-native artifact**: authoring and validation run on Claude, and the skill the user owns is an [Agent Skills open-standard](https://agentskills.io) `SKILL.md` folder that installs as-is across compatible tools (Claude, Codex CLI, Gemini CLI, Copilot and more). **Honest portability** is the growth edge ‚Äî validating *behaviour* on other runtimes' models, never a "runs everywhere" fidelity claim; the standard already solved the format question.

**Audience is a bridge, not just developers.** Two users:

- the **technical builder** ‚Äî fluent in `SKILL.md`, YAML, trigger logic;
- the **non-technical SMB owner** ‚Äî automating admin (inbox, scheduling, docs) with AI.

A skill is an inherently technical artifact (instructions + frontmatter + trigger logic), so the product's job is to make it *approachable without dumbing it down*: **one warm-pro identity, technical depth on demand, not in your face.** This is a positioning commitment, not just styling ‚Äî it drives the hero's [Rendered/Source render model](#7-frontend--app-shell) and the whole `DESIGN.md`.

**The wedge is the hosted graphical flow.** The eval tooling that exists elsewhere is CLI-bound (Anthropic's skill-creator runs trigger evals as a terminal meta-skill), so for the builder who will never run a CLI, a hosted surface ‚Äî evaluations launched from the hero's chips, landing as Insights ‚Äî is the only validation on offer. That gap, plus **honest cross-runtime results** ([#116](https://github.com/CrowBe/agentbranch/issues/116)) and **honestly-labelled distribution** ([¬ß9.1](#91-skill-tap--open-distribution-with-safety-badges)), is the defensible ground; a triggering eval alone is table stakes the CLI tools already provide. **Teams** (shared libraries, private validation gates, seats) are the named later revenue layer, not a v1 audience.

**v1 = a thin vertical slice across all capabilities, Claude-only.** The authoring loop ships polished; everything else ships thin. Full breadth, minimum depth.

## 2. Glossary ‚Äî the domain language

One term per concept. Use these names everywhere ‚Äî in docs, code, and UI copy.

| Term | Definition |
|---|---|
| **Skill** | The product's unit of work: a reusable instruction set for an AI agent, per the **Agent Skills open standard**. Instruction-only ‚Äî no bundled runnable code. Claude is the first-class authoring and validation runtime; the artifact itself is standard-native and installs across compatible tools. |
| **`SKILL.md`** | The skill's source file (the open standard's format): YAML frontmatter (`name`, `description`) + markdown body (instructions, workflow, rules). |
| **Skill record** | The persisted skill in our DB (see [¬ß6](#6-data-model-sketch)). Exports are rendered *from* it. |
| **Skill metadata** | A skill's discovery metadata: one **category** from a closed taxonomy + up to 8 kebab-case **tags**, living in `SKILL.md` frontmatter (so it travels with the standard-native artifact and is pinned by the content hash). Drives Skill-library filtering/search and the profile pages' "class" grouping. Authored as a mix ‚Äî build-loop drafting writes it, the author edits it, the **metadata-suggest** capability (analysis, model-backed with an offline fallback) recommends it. |
| **Build loop** | The core agentic loop: Claude writes/edits the `SKILL.md` through the `write_skill`/`edit_skill` tools, streaming to the preview. Reaches the model only through the **model gateway** (`streamAgent`), so a build turn is gated + accounted like every other model call. The loop is *closeable* with evaluation and lint results: an **eval feedback** message can be appended to the conversation, giving Claude the specific failure cases and rationale it needs to make targeted revisions. |
| **Eval feedback** | A formatted summary of an **Evaluation result** (or **Lint** artifact) injected as a user message into the build loop conversation, so Claude can revise the skill against observed evidence rather than guessing. Produced by a **feedback formatter** ‚Äî a pure function in the `build-loop` module that translates the structured artifact into actionable revision context. User-triggered from the **Insights** surface for evaluation results ("Revise with this feedback"); auto-injected after `write_skill` for lint findings. Distinct from **Insights** (the user-facing display surface) ‚Äî Insights is what the user reads; eval feedback is what Claude reads to author the revision. |
| **Skill-analysis seam** | The architectural spine. The shared pattern **read input ‚Üí emit a structured artifact ‚Üí render it for a surface**. Built once; each feature is a new capability on it, not a new pipeline. Carries **two capability shapes** (below). Distinct from the skill IR: the seam is the pattern, the IR is one artifact type on it. The `Analyzer<Input, A>` / `Evaluator<Input, A>` input slot is generic: `Skill` was the first concrete input, and the **equipment primitives** (response schemas, tool contracts, [¬ß9.2](#92-equipment-primitives--composition-broadening-beyond-skill)) ride the same seam. |
| **Analysis capability** | A *static* capability on the seam ‚Äî derives a structured view from an input's source. Pure, runs offline. Wraps an `Analyzer<Input, A>` (`analyze(input)`). The Rendered/Source hero, Visualise, Lint and Export are analysis capabilities over `Skill`; Response schema quality and Tool contract quality are analysis capabilities over their own primitives. |
| **Evaluation capability** | A *dynamic* capability on the seam ‚Äî **runs an input through a model** and observes behaviour. Costs tokens, needs a model (fails `model_unavailable` offline). Wraps an `Evaluator<Input, A>` (`evaluate(input, gateway)`) that **owns its method, not its resources**: it builds its own conditions (Scenario / distractors / battery) but model access is handed in via the **model gateway**. Triggering eval is an evaluation capability over `Skill`; Test run's input is a **bundle** ‚Äî the Skill, optionally with Tool contracts + Response schemas. |
| **Evaluation result** | The artifact an evaluation capability emits ‚Äî the structured run-record. Ephemeral; **never shown raw**. Renders to **Insights** (and a detailed breakdown on demand). Distinct from the persisted **evaluation record** ([¬ß6](#6-data-model-sketch)): the result is rendered now, the record is stored and re-rendered later. |
| **Insights** | The default, plain-language rendered surface of an evaluation result ‚Äî meaning the user can act on, not a data wall. The audience bridge ([¬ß1](#1-product-thesis)) lives in the *renderer*; a detailed breakdown sits behind it for depth (same Rendered/Source duality as the hero). |
| **Model gateway** | The platform's **single, controlled, metered entry to the model** ‚Äî its own module. Owns the AI-SDK call plumbing; exposes fine intent-level primitives `classify`/`runAgent`/`streamAgent`/`generate` that callers compose into their own method. Pure mechanism ‚Äî knows no capability kinds. It does **not** pick the provider/model or hold the key: it resolves a `LanguageModel` per call from the **model router**. Every call carries an **accounting tag** (`account` \| `platform`); the gateway routes accounting through the **usage** authority. Consumers: the build loop (via `streamAgent`) and evaluation. Cross-runtime validation uses its per-call target selection; mock-data generation also routes through it. **Target state ([#34](https://github.com/CrowBe/agentbranch/issues/34), [#35](https://github.com/CrowBe/agentbranch/issues/35)):** as the one metered chokepoint, the gateway is also where input-size budgets, prompt caching (a frozen system prefix), and cache-aware token accounting are enforced and measured. |
| **Model router** | The platform's **single provider + model selection authority** ‚Äî its own module, the layer beneath the gateway. Owns the **provider registry** (every provider the platform knows: Anthropic default, Nous Portal, extensible), their **credentials** (a server-pool key per provider, plus an optional **bring-your-own override** that wins when present), and the runtime-mutable **active selection** (which provider + model each primitive routes to). The gateway calls `resolve(primitive)` to get a `LanguageModel`; the **model console** drives the router's mutators so provider/model can be switched (or a key connected) at runtime, e.g. to rotate providers while testing the flow. Pure *selection* mechanism ‚Äî knows no accounting or capability kinds. Per-capability routing (Opus for `streamAgent`, cheaper models for `classify`/`generate`) is expressed here as per-primitive model ids. Selection + bring-your-own keys are **process-local and single (not per-user)** in v1 ‚Äî one shared active selection per instance, which suits the testing/rotation use case; per-user isolation + a persisted store is a future port + adapter swap. Keys never leave through its secret-free snapshot and are never logged. |
| **Accounting tag** | Declared by the *caller* on every model-gateway call: **`account`** (user-attributable, subject to tier policy) or **`platform`** (the platform's own cost to enable a feature ‚Äî e.g. generating mock data ‚Äî never charged to a user's allowance). Three accounting streams: free+account = structural caps + provider cap-catch (no token counting, [¬ß8](#8-free-tier)); paid+account = token-spend stream (deferred); platform = our own cost ledger (deferred). A model call denied by a cap fails `cap_reached` (distinct from `model_unavailable` = no model at all). |
| **Skill IR** | *One specific* artifact type on the seam: visualise's intermediate representation ‚Äî nodes + edges, each carrying a **source-span** back into `SKILL.md`. A visualise concept, not the whole seam. |
| **Test run** | User-facing term for executing a skill against **mocked** tools to see how it behaves. The mechanism is the **mock-tool registry**; the agent tool is **`execute_skill`**. Nothing real is ever touched. (Always "test run" in user copy ‚Äî never "sandbox": jargon that intimidates rather than informs.) |
| **Mock-tool registry** | The mechanism behind a test run: when the skill calls a tool (e.g. "fetch unread email"), the registry returns generated mock data instead. The skill runs end-to-end against fake tools, so the user sees its behaviour without anything real happening. No code execution or containers ‚Äî skills are instruction-only, so there's nothing to containerise. |
| **Response schema** | The first equipment primitive beyond Skill ([¬ß9.2](#92-equipment-primitives--composition-broadening-beyond-skill)): a structured output definition ‚Äî a JSON Schema document ‚Äî with a lossless source model and pure offline lint. No evaluator; its schema subset also validates tool-contract examples and test-run calls. Chat-buildable through the build loop's shape under its own frozen authoring prompt ([¬ß9.2](#92-equipment-primitives--composition-broadening-beyond-skill)). |
| **Tool contract** | The second equipment primitive: a tool's typed input/output plus description, examples, failure modes, and safety notes. Its I/O is an inline schema or a `$ref` to a **response schema** by title. Standalone it is an analysis primitive (lossless source model + lint) and chat-buildable through the build loop's shape under its own frozen authoring prompt; in a **bundle** it drives the test run's mock tools and per-call validation. |
| **Bundle** | The test run's composed input ([¬ß9.2](#92-equipment-primitives--composition-broadening-beyond-skill) *smallest useful composition*): the Skill plus selected Tool contracts and the Response schemas they reference. Contracts drive the mock-tool registry (mock output conforms to the contract's output schema) and each observed call is validated ‚Äî arguments against the input schema, handled output against the output schema. |
| **Triggering eval** | The v1 validation: does the skill *fire* on the right prompts and *stay silent* on the wrong ones? Run competitively against a **distractor library** + a positive/negative **prompt battery**. |
| **Safety rating** | *User-facing term* for the persisted result of a **safety review** run ([¬ß9.1](#91-skill-tap--open-distribution-with-safety-badges)): verdict (passed / needs review / blocked) + per-class risk scores + Insight, pinned to the skill version reviewed. **Opt-in and manual, never automatic** ‚Äî offered any time for an unrated version and as an optional, non-gating step of "Set as main version". A passing rating pinned to a published version's hash earns the **safety badge**; a published version without one is labelled *potentially unsafe ‚Äî not validated*. Runs spend `account` (`safety-review` capability); the capability stays caller-tagged. |
| **Cross-runtime validation** | The portability surface. Skills travel as-is under the open standard, so portability is a *behaviour* question, not a format one: one engine runs the skill's triggering battery against selected runtime targets through the model gateway and reports a per-target grid. Results are honest, model-level approximations of each tool's runtime ‚Äî copy says so. |
| **Rendered / Source view** | The hero's two views of the same skill. **Rendered** (default) = friendly sans-serif document. **Source** = raw `SKILL.md` monospace. Both are renderers on the seam. |

## 3. Core architecture ‚Äî the through-line

A single **server-side agentic harness** (Vercel AI SDK + Claude) with a **tool registry**. Not N services ‚Äî one agent loop, many tools.

```
Browser (React)  ‚îÄ‚îÄSSE‚îÄ‚îÄ‚ñ∂  Next.js route handler
       ‚ñ≤                          ‚îÇ
       ‚îÇ stream tool output       ‚ñº
   preview / viz / chat     Build loop  ‚îÄ‚îÄstreamAgent‚îÄ‚îÄ‚ñ∂  Model gateway ‚îÄ‚îÄresolve‚îÄ‚îÄ‚ñ∂ Model router
                                  ‚îÇ                      (metering + accounting)    (registry + creds
                                  ‚ñº                              ‚îÇ                   + active selection)
                            Tool registry:                       ‚ñº                          ‚îÇ
                              write_skill, edit_skill,     Claude / Nous (Vercel AI SDK) ‚óÄ‚îÄ‚îÄ‚îò
                              visualise_skill, execute_skill,                  ‚ñ≤
                              + mock-tool registry integrations          Model console (UI)
```

- The **model router owns the provider keys** + the active provider/model selection; the **gateway** owns metering + accounting. Nothing above the gateway (the build loop, the route handler) touches the raw model, the keys, or the client. The **model console** is the only surface that mutates the selection, through the gateway-adjacent router.
- Tool-call output streams to the client over **SSE**. Evaluation runs use the same transport for progress events and the final artifact.
- `execute_skill` runs against the **mock-tool registry** ([¬ß2](#2-glossary--the-domain-language)): a test run, not real execution.

### 3.1 The skill-analysis seam (the spine)

Several capabilities are the **same shape**: *read an input ‚Üí emit a structured artifact ‚Üí render it for a surface*. Build the seam once; each feature plugs in without a new pipeline. The generic `Input` slot is load-bearing: most capabilities read a `Skill`, the equipment primitives (response schemas, tool contracts, [¬ß9.2](#92-equipment-primitives--composition-broadening-beyond-skill)) read their own source models, and the test run reads a **bundle**.

The seam carries **two capability shapes** ‚Äî same `artifact ‚Üí render` tail, different head:

- **Analysis** (static) ‚Äî derives an artifact from the input's source. Pure, runs offline. Wraps an `Analyzer<Input, A>` (`analyze(input)`). Its artifact is *recomputed on demand*, never persisted. Rendered hero, Source view, Visualise, Lint and Export use `Skill` as their input; Response schema quality and Tool contract quality use their primitive's source model.
- **Evaluation** (dynamic) ‚Äî produces an artifact by **running the input through a model**. Needs a model (fails `model_unavailable` offline), costs tokens. Wraps an `Evaluator<Input, A>` (`evaluate(input, gateway)`) that **owns its method, not its resources** ‚Äî it builds its own conditions but is handed the **model gateway**, the platform's single metered entry to the model. It composes its method from the gateway's fine primitives (`classify`/`runAgent`); it never touches the raw model, the key, or token accounting. Its artifact (an **evaluation result**) is *persisted* as an evaluation record and renders to **Insights**. Triggering eval uses `Skill` as its input; Test run uses a **bundle** (the Skill, optionally with Tool contracts + Response schemas).

> The split is named so a new capability is classified **before** it's built: "is this analysis or evaluation?" decides whether it gets an `Analyzer` or an `Evaluator`, runs offline or needs the harness, recomputes or persists. The seam stays one spine; the two heads keep the model dependency and token cost out of the pure analysis surfaces.

| Capability | Shape | Extract / run | Render (v1) | Render (later) |
|---|---|---|---|---|
| **Rendered hero** | analysis | frontmatter + body | friendly structured document (sans-serif) | richer doc layout / inline editing |
| **Visualise** | analysis | skill IR (nodes+edges+source-spans) | Mermaid | interactive canvas (React Flow) |
| **Lint** | analysis | frontmatter + body quality rules ‚Üí `LintReport` | Insights (score/grade + actionable findings) + Breakdown (full finding list with source-spans) | per-primitive rule sets as new equipment types land |
| **Response schema quality** | analysis | JSON Schema document structure + smell rules ‚Üí LintReport-shaped artifact | Insights + Breakdown (same quality vocabulary as Lint) | referenced-schema graph checks |
| **Tool contract quality** | analysis | I/O typing completeness, description/example quality, failure modes, safety notes ‚Üí LintReport-shaped artifact | Insights + Breakdown | resolve `$ref`s against a schema library |
| **Cross-runtime validation** | evaluation | run triggering battery on selected runtime targets ‚Üí per-target results | Insights | per-target results grid |
| **Test run** | evaluation | run the bundle vs. mock-tool registry (contract-driven when contracts are supplied) ‚Üí result + per-call contract checks | **Insights** (+ transcript & contract checks on demand) | richer scenarios / multi-run |
| **Triggering eval** | evaluation | run skill vs. distractors + battery ‚Üí result | **Insights** (+ pass/fail breakdown) | scored / workflow / regression |

This is what "build it flexible" means here: a **clean seam**, not a speculative v2 abstraction.

## 5. v1 thin-slice spec (per capability)

The build loop is **not thin** ‚Äî it's the core and must feel good. Everything else is the minimum that proves the shape.

1. **Auth + meter** ‚Äî Clerk (Google+GitHub), Postgres, per-turn + per-token counter. Thin = no admin UI, enforced caps only.
2. **Build loop** ‚Äî Claude via the model gateway's `streamAgent`; `write_skill`/`edit_skill`; SSE to preview. **Polished, not thin.** Opens with the **requirements interview** ([¬ß4](#4-decisions)) before the first draft. Closeable with **eval feedback** ‚Äî triggering-eval and test-run Insights feed back as messages, giving Claude the failure cases and rationale to make targeted revisions.
3. **Lint** ‚Äî pure analysis, zero tokens, uncapped; auto-runs on every version after `write_skill`. Insights + Breakdown renderers. The seam's `Analyzer<Input, A>` design makes lint rule sets portable to future equipment primitive types.
4. **Import** ‚Äî paste `SKILL.md` text or import a public GitHub skill URL; same parse ‚Üí persist ‚Üí render path as authored skills.
5. **Visualise** ‚Äî model emits skill IR; thin IR‚ÜíMermaid renderer; 1 generation; no diagram editing.
6. **Test run** ‚Äî single run, 1 generated scenario, mock-tool registry with 1‚Äì2 mock integrations (email first), Claude only.
7. **Triggering eval** ‚Äî corpus-fed distractor library (~20), small prompt battery, "did it fire?" boolean. No judge model yet.
8. **Export** ‚Äî copy + the standard skill folder `.zip`.
9. **Cross-runtime validation** ‚Äî validation-first engine: run the triggering battery against selected runtime targets through the model gateway, with unconfigured targets shown honestly.
10. **Billing** ‚Äî Clerk subscription tiers (Free + one Pro), gating capabilities 6/7. Thin = no PAYG.

## 6. Data model (sketch)

- **`users`** ‚Äî identity from Clerk (Google/GitHub).
- **`skills`** ‚Äî `id, user_id, name, description, body, frontmatter_json, main_version_id, created_at`. `main_version_id` is the **explicit blessed pointer** ‚Äî the version everything defaults to (export, install, the hero's first paint). On the linear path it tracks the newest revision; branching makes it explicit and independent of newest, so a draft can append higher revisions without becoming main. *Main is just the branch that owns this version.*
- **`skill_branches`** ‚Äî a line of iteration (a **draft** in user copy, [¬ß9.3](#93-branching-iteration--draft-main-version-promote)): `id, skill_id, status (open | discarded), ordinal`. Every version belongs to one; the branch owning the skill's `main_version_id` *is* the blessed lineage, so `isMain` is derived, never stored. A skill carries several open drafts at once.
- **`skill_versions`** ‚Äî append-only revisions of a skill (export is a pure function of a version; enables future regression evals). Versioning is **draft / main / promote** ([¬ß9.3](#93-branching-iteration--draft-main-version-promote)): a version belongs to a branch and chains by `parent_id` (a DAG), so two drafts advancing independently never collide. `revision` is a **per-branch display ordinal** (`@@unique([branch_id, revision])`), not a per-skill monotonic counter. **Restore** lands a copy as a new head revision and **promote** re-points `main_version_id` to a draft's head ‚Äî both append-only (no version is mutated or deleted). Retention is **off the write path** ‚Äî a **daily cleanup job** (the `SkillRetentionRepository` port + a Vercel Cron trigger, [¬ß9.3](#93-branching-iteration--draft-main-version-promote)) enforces latest-N-per-draft depth *and* an **open-drafts-per-skill cap**, never pruning the main lineage or an open draft's tip. Pruned draft versions release their pins via `onDelete: SetNull`, so stored run records survive.
- **`usage`** ‚Äî per-user token + turn counters (drives caps now; drives PAYG metering later).
- **`publications`** ‚Äî public addresses for skill versions ([¬ß9.1](#91-skill-tap--open-distribution-with-safety-badges)): publisher, `owner/skill-name` slug, trust tier (private / published / reviewed), pinned append-only `skill_version_id`, and content hash. Publishing is ungated; the version's safety rating (or its absence) drives the badge/flag presentation, and a changed version means a different hash ‚Äî unbadged until re-analysed.
- **`harness_versions`** ‚Äî append-only identity for the validation harness itself: one manifest hash decomposed into sub-hashes for the build-loop system prompt, lint ruleset, prompt-battery generator, test-run world generator, distractor library, plus the deploy git SHA when available. Source artifacts stay in git; the row records identity + hash so outcomes are attributable to the harness that produced them.
- **`eval_runs`** / **`test_runs`** ‚Äî recorded triggering-eval and test-run results. Each run carries nullable `skill_version_id` and `harness_version_id` pins: the skill revision evaluated and the harness manifest that produced the result. Both use `onDelete: SetNull`, so stored runs survive version cleanup or harness-retention changes. List/fetch APIs plus My skills / History surfaces re-render stored records through the seam renderers.
- **`safety_ratings`** ‚Äî recorded **safety ratings** ([¬ß9.1](#91-skill-tap--open-distribution-with-safety-badges)): verdict + per-class scores + Insight for one safety-review run, with the same nullable `skill_version_id` / `harness_version_id` pins (`onDelete: SetNull`) as the other evaluation records. The version pin is load-bearing twice over: "does this version carry a rating?" is a `latestForVersion` read, so an unrated head is what triggers the opt-in offer ‚Äî and the same read is what decides whether a published version renders the safety badge or the potentially-unsafe label ([¬ß9.1](#91-skill-tap--open-distribution-with-safety-badges)).
- **`benchmark_runs`** ‚Äî scorings of the frozen regression-benchmark set, each pinned to the harness version in effect and the set hash it scored ([¬ß9 *Harness improvement loop*](#harness-improvement-loop-admin)). Append-only; admin-only surface, so no user scoping.

**Persistence invariants.** All DB access goes through Prisma's query builder ‚Äî **no raw SQL anywhere**, so stored skill content can never become a SQL payload (parameterized-only). ESLint bans raw SQL calls unless a reviewed exception is added. Row ownership is enforced *in the query* (`where: { id, userId }`) for user-owned records rather than only re-checked in callers. **Target state ([#34](https://github.com/CrowBe/agentbranch/issues/34)):** `frontmatter_json` bounded (size / depth / key-count) with unsafe keys (`__proto__`, `constructor`) rejected on parse. Skill content and prompts are never written to logs.

Export shapes are **rendered from** the skill record ‚Äî the schema is export-agnostic.

Presentation-layer architecture ‚Äî same kind of decision as the rest of this doc (it determines components, shell, routing), so it lives here.

First-run needs a one-time hint so the expandability is discoverable.

## 9. Deferred features & their seams

Designed-for, not built. Each reuses an existing seam, so it's a renderer/config swap ‚Äî not a new pipeline. The two principles worth stating explicitly:

- **The skill-analysis seam is the spine** ‚Äî visualise, the Rendered hero, export, lint, and triggering evals all plug into it. New capabilities should ask "what input, artifact, and renderer is this?" before "what service is this?"
- **Cross-runtime validation is an evaluation capability on the seam** ‚Äî a provider swap inside one engine, not a parallel pipeline.

**Build order after the v1 slice** ‚Äî with the cross-runtime validation engine in place, the differentiators continue with the **draft/promote UI** ([#129](https://github.com/CrowBe/agentbranch/issues/129)) riding the already-landed branch substrate, then the **skill tap's validation stack and publication substrate** ([#133](https://github.com/CrowBe/agentbranch/issues/133) ‚Äî shipped; the outward-facing distribution half remains). Primitives phases 4‚Äì6 ([#126](https://github.com/CrowBe/agentbranch/issues/126)) wait for a demand signal. The **harness improvement loop** ([#118](https://github.com/CrowBe/agentbranch/issues/118)) is built end-to-end (see [*Harness improvement loop*](#harness-improvement-loop-admin) below); its recommendations sharpen as the corpus of real user runs grows.

**Deferred capabilities:**

- **Branching iteration** *(built ‚Äî the one shipped item in this list; kept here because [¬ß9.3](#93-branching-iteration--draft-main-version-promote) is its design home)* ‚Äî safe-space iteration: a user edits and evaluates a **draft** without disturbing the version they trust, then **sets the draft as the main version** when satisfied. Several drafts can be open at once, so an un-promoted draft from a prior session is never stranded. Borrows git's *shape* without forcing git vocabulary onto the SMB-facing UI. Both halves have landed: the **data substrate** ([#128](https://github.com/CrowBe/agentbranch/issues/128)) ‚Äî branch/promote on `SkillRepository` + the daily retention job ‚Äî and the **evaluate-on-draft + promote surface** ([#129](https://github.com/CrowBe/agentbranch/issues/129)) ‚Äî draft edits build on the branch, evaluation records pin the draft head, and the hero makes the draft-vs-main state legible with **Set as main version** / **Discard draft** controls. Full design in [¬ß9.3](#93-branching-iteration--draft-main-version-promote).
- **Cross-runtime validation** *(built engine; richer surface later)* ‚Äî the existing triggering battery run against selected runtime targets through the model gateway; returns per-target results and an Insight, with unconfigured targets marked honestly.
- **Interactive visualise canvas** ("Claude design"-style point-and-annotate) ‚Äî IR‚ÜíReact Flow renderer; **point-and-annotate falls out for free**: node ‚Üí its source-span ‚Üí inject that span as precise context into the next chat turn.
- **Richer evals** ‚Äî workflow evals (run the full loop, LLM-as-judge against rubrics) ‚Üí regression evals (pin a scenario set, track score across edits = the retention hook).
- **Baseline skill corpus** ([#149](https://github.com/CrowBe/agentbranch/issues/149)) *(built)* ‚Äî a repo-tracked set of 20 curated skills for common moderately-technical and non-technical workflows (email triage, calendar planning, meeting notes, and the like), with source prompts, prompt batteries, provenance, and content hashes. It *is* the frozen regression benchmark's set (below), feeds the triggering eval's distractor library, and is ready to seed Templates.
- **Insight agent** ‚Äî today each evaluation produces its **Insight** (the plain-language interpretation rendered as Insights) from one bounded `generate` call. The richer future is a tool-using agent (`runAgent` + tools) that *investigates* a result ‚Äî re-runs cases, inspects the skill ‚Äî before explaining it. Cross-cutting: it would explain *any* evaluation kind, reusing the seam, and replaces the `generate` call without changing the `Insight` shape or the renderer.
- **Skill tap** (publish + Skill library) *(validation stack, publication domain, distribution read models, publish route, tap repository snapshot, workspace surfaces, and source-to-tap sync script built; the public tap repo and its secrets remain)* ‚Äî a public installation source other tools add once and install from. Publishing is open; validation is offered as an opt-in **safety badge**, and unvalidated published versions are labelled potentially unsafe ‚Äî full design in [¬ß9.1](#91-skill-tap--open-distribution-with-safety-badges) ([#133](https://github.com/CrowBe/agentbranch/issues/133)).
- **Own-portfolio collision detection** ‚Äî once accounts hold many skills, eval a new skill against the user's existing ones for trigger overlap.
- **PAYG / metered overage** ‚Äî Stripe Billing Meters on the existing token meter; transparent margin. Pairs with **separate Anthropic keys/workspaces per tier** the moment paid users exist, so free-tier exhaustion can't block paying customers.
- **Realtime voice interaction** ‚Äî the interaction surface evolves from the slim typed drawer (v1) to a floating **voice-forward control** (mic primary, typing fallback, no transcript column). Preview-primary was chosen so this is an *evolution, not a rework*; v1 UI choices must not block it. Slim drawer will stay as rendered transcript even in v2.

### Harness improvement loop (admin)

Built ([#118](https://github.com/CrowBe/agentbranch/issues/118)). The harness improvement loop is the admin-only feedback system that reads stored evaluation records across sessions and recommends changes to the validation harness itself. It is the first use of the skill-analysis seam where `Input` is not a single `Skill`: input is a **cohort of evaluation records** (triggering eval + test run records, pinned to skill versions and harness versions), artifact is a **harness-recommendation report**, and renderer is an **admin report surface** (the `harness-recommendation` module; `/api/admin/harness-report`). Tier 1 is static correlation, so the capability is an **analysis capability** ‚Äî zero tokens, runs offline; a future model-written tier would be an evaluation capability (`defineEvaluation`) spending with a `platform` accounting tag.

The recommendation targets are the versioned pieces of the harness, not arbitrary product text: the build-loop system prompt, lint rules, test-run data-generation generator, triggering-eval prompt-battery generator, and distractor library. For the two generators, the thing attributed and improved is the **generator prompt + schema**, not any one generated scenario or prompt battery output.

Attribution needs two axes: **skill version √ó harness version**. The skill version says what artifact was evaluated; the harness version says which build-loop prompt, lint rules, generator prompts, schemas, and distractors produced the outcome. Without both pins, a later report cannot tell whether a failure came from the skill, the harness, or a changed generator. The `harness_versions` table is the second axis; stored `eval_runs` and `test_runs` carry nullable pins to both so old records survive retention.

The one aggregate persistence surface is the **admin aggregate read** on the eval/test-run repositories (`listForAnalysis`, Prisma + memory). User-scoped reads remain the default and ownership stays enforced in query for ordinary product surfaces; the aggregate read is exposed only behind `isAdmin` and the same allowlist posture as the model console. Its read model is outcomes and features by construction ‚Äî pass/fail, case expectations and rationales, fired lint rules (joined from the skill version's stored lint summary), timestamps, skill and harness ids ‚Äî user identity, prompt text, and `SKILL.md` content never cross into it. Reading content would require an explicit consent boundary because cross-user content is sensitive even when the goal is product improvement; no such read exists.

The first recommendation tier is deliberately auditable: correlate the lint rules a skill version fired with its triggering-eval outcomes and propose lint-rule changes ‚Äî reweight a rule whose presence tracks failure, review one that fires mostly on passing skills, mine lint-clean failures for a new rule ‚Äî each traceable to its evidence run ids. A human applies any change as an ordinary reviewed diff; the report never mutates anything. The **regression benchmark** is the guardrail: the frozen set (the baseline skill corpus + its curated batteries, pinned by a set hash) is scored through the triggering eval's competitive selection and recorded per harness version (`benchmark_runs`, `/api/admin/benchmark`), so vN+1 must beat vN on identical ground before a recommendation counts as an improvement. This is the cross-session cousin of **eval feedback**: eval feedback helps Claude revise one skill in the build loop; the harness improvement loop helps admins improve the evaluators and prompts that produce those results.

### 9.1 Skill tap ‚Äî open distribution with safety badges

The in-app half is built and the public half has its backend snapshot ([#133](https://github.com/CrowBe/agentbranch/issues/133)). Built: the full validation stack the badge rides on (lint policy rules, adversarial negative battery, `safetyReviewCapability`), the publication domain (`publications`, [¬ß6](#6-data-model-sketch)), the publish route, the pure distribution read models and file renderers ‚Äî the tap marketplace manifest, the standard skill folders for the tap repo, the tap repository file-set route (`/api/tap-repository`), and the Skill-library feed (`/api/skill-library`, Templates as a view over the reviewed tier) ‚Äî the workspace Templates / Skill library surface that consumes the feed with search, direct slug lookup, badge/flag, trust tier, content hash, and footprint copy at the install boundary, and a source-to-tap sync script that applies the snapshot to a tap repo checkout for the bot PR flow. Not built: the public tap repo itself, its advisory CI, and the production bot wiring that opens and merges PRs. "Tap" (an installation source you add once, then install from by name) is the internal term; user-facing copy says **Publish** / **Skill library**. A tap is *distribution*, not deploy: a published skill gets a public address, nothing runs.

**Publishing is open ‚Äî validation is offered, never enforced.** No moderation step gates a publish. Instead, honesty does the work: a published version that carries a passing **safety rating** (below) pinned to its exact content hash renders a **safety badge**; a published version without one is labelled **potentially unsafe ‚Äî not validated**. A changed version means a changed hash, so a revision is simply unbadged until the owner re-runs the analysis ‚Äî the badge can never drift from the content it reviewed. Everything the old gate would have run stays available as the badge's substance and as authoring-time guidance (lint policy findings, adversarial battery, test run); none of it blocks distribution.

**Threat model.** Skills carry no code, but that bounds *our* platform, not the consumer's runtime ‚Äî a published `SKILL.md` installs into agents holding shells, browsers, and credentials. Model a skill as **a program whose interpreter is an LLM**. Classes: (1) **direct malicious instructions** (the skill is the payload, the host agent the executor); (2) **agent hijacking** ‚Äî overriding the host's guardrails; (3) **trigger hijacking** ‚Äî a deceptive `description` fires the skill where it has no business, the tap-specific risk since the standard selects on descriptions; (4) **latent payloads** ‚Äî benign on read, malicious on a date/keyword/fetched URL; (5) **reference-file smuggling** ‚Äî every gate layer reads the whole folder, not just `SKILL.md`; (6) **typosquatting / impersonation**. Hard limit, accepted: class 4 isn't solvable by review ‚Äî anything checked once is defeated by indirection. This threat model is why the honesty labelling exists: it is the live rationale for the badge/flag split and the install-boundary copy. *Content policy* remains its one static countermeasure ‚Äî instructing the agent to fetch-and-follow remote instructions is a hard lint failure ‚Äî but it is authoring guidance and badge substance, not a publish gate: a version tripping it can be published (flagged), it just can't be badged.

**The badge is the seam** ‚Äî agent.branch is a validation tool that also distributes, not a file host bolting a rating on. The badge's substance reuses the seam: **skill lint** carries the static policy rules (shell-exec patterns, credential paths, fetch-and-follow indirection, obfuscation), reading the whole folder when reference-file contents are provided; the **triggering eval**'s negative battery includes fixed adversarial prompts, making it the trigger-hijack scanner; the **test run** catches behaviour the text hides (exfiltration-shaped tool calls show in the transcript); and the **`safetyReviewCapability`** ‚Äî an LLM-judge evaluation capability, caller-tagged ‚Äî reads the full folder and scores injection/exfiltration/deception. The judge is **structurally untrusted**: it reads the skill strictly as data, and its verdict is a claim about the artifact, never a guarantee about a runtime.

**Safety ratings are opt-in ‚Äî manual, never automatic ‚Äî and earn the badge.** The `safetyReviewCapability` surfaces as a **safety rating** the user runs on demand. A rating pins the version it reviewed; versions are append-only, so a rating stays true forever and a new revision is simply *unrated* ‚Äî the scan is offered any time for an unrated version (the hero's **Safety** chip) and as an **optional step of "Set as main version"** (skippable, and a worrying verdict only asks ‚Äî it never blocks promote or publish). A published version whose head carries a passing rating renders the **safety badge**; an unrated or non-passing published version carries the **potentially unsafe ‚Äî not validated** label instead. Because the user asks for it, the run is `account`-tagged against the `safety-review` gated capability (Pro tier) ‚Äî nobody burns credits on a rating they didn't request; the capability stays caller-tagged, so any future platform-initiated review would spend `platform`. A rated version never re-spends: reopening the rating and switching Insights/breakdown render the stored record. Ratings persist as `safety_ratings` ([¬ß6](#6-data-model-sketch)), pinned to skill version + harness version like every evaluation record.

**Tiered visibility ‚Äî open existence, humans for amplification.** Effort scales with what we promote, not with volume:

| Tier | Bar | Reach |
|---|---|---|
| **Private** | ‚Äî | Default; owner only (status quo). |
| **Published** | None ‚Äî publishing is open | In the tap, link-reachable, installable. Carries the **safety badge** (passing rating pinned to this hash) or the **potentially unsafe ‚Äî not validated** label. |
| **Reviewed** | Published **+ human review** | Surfaced: search, Templates, featured. What we vouch for ‚Äî an editorial pick, not a moderation verdict. |

Supporting policy: publishers are attributable (OAuth-only identity, already true); publish pins an append-only skill version + **content hash** (a badge provably describes the thing served; new version = unbadged until re-analysed); publish attempts are **rate-limited** via the existing per-capability windows; slugs live under the publisher's handle (`owner/skill-name`), removing most squatting value.

**The tap is a public git repo ‚Äî everything visible, nothing gated.** The repo carries a `.claude-plugin/marketplace.json` (add once in Claude Code, install by name) and **is the publication mechanism**, Homebrew-style: publish = bot PR adding the rendered standard skill folder pinned to a version + hash, auto-merged ‚Äî no check blocks it. The **lint policy rules still run as open-source CI in the tap repo** and the hosted layers still post verdicts to the PR, but as **advisory annotations** feeding the badge/flag metadata, not merge conditions. The **reviewed** tier is a curated index changed only by human-approved PRs, so vouching is itself a public, attributable diff; **takedown = revert at HEAD** (installs read HEAD, so a revert ends installability immediately ‚Äî the abuse backstop that replaces up-front gating), with reports via repo issues and an in-app path. Accepted residuals, named: git history persists (revert immediately, purge via host support for confirmed malware ‚Äî tolerable for instruction text where it wouldn't be for binaries); static rules are public (attackers can test offline; transparency buys more trust than the obscurity protects); and **unbadged malicious content can be live until reverted** ‚Äî the model accepts detection-and-takedown plus labelling in place of prevention, which is why the flag copy must be blunt.

**Consumer-side honesty.** We can't make another runtime safe and the standard has no permissions manifest. At the install boundary we surface what analysis can derive: the skill's tool/behaviour footprint, trust tier, the safety badge or the potentially-unsafe label, rating results when they exist, content hash. Presentation, never a guarantee ‚Äî copy says so.

**Public skill profile pages.** Every visible publication gets a public page at `/skills/<owner>/<name>` ‚Äî the discovery/SEO surface of the tap. The page is a pure render over the pinned version and its safety state (profile read model + the hero capability's `rendered` view), served force-dynamic so a takedown revert or a new rating is visible immediately; it can never describe content it doesn't match, for the same reason the badge can't drift. Category and tags from the pinned frontmatter drive the Skill library's `?category=` / `?tag=` filters and tag-aware search; the same metadata is how "top skill in its class" curation gets its classes ‚Äî curation itself stays the human-approved reviewed tier, never a score leaderboard.

**Build order:** the validation stack (static lint policy, adversarial negative battery, `safetyReviewCapability` + opt-in safety rating), the publication domain (pinned version, slug, tier, hash, rate limit), the distribution read models (tap marketplace manifest, tap repository file set, Skill-library feed with Templates as a view over the reviewed tier), the publish route, the workspace Templates / Skill library surface, and the source-to-tap sync script all run today. Then the outward-facing half: provision the public tap repo, advisory CI, and production bot wiring ([#173](https://github.com/CrowBe/agentbranch/issues/173)).

### 9.2 Equipment primitives & composition (broadening beyond Skill)

Tracked in [#126](https://github.com/CrowBe/agentbranch/issues/126) (paired with branching iteration, [¬ß9.3](#93-branching-iteration--draft-main-version-promote)). **The first two primitives and the first composition are built** ([#130](https://github.com/CrowBe/agentbranch/issues/130)‚Äì[#132](https://github.com/CrowBe/agentbranch/issues/132)); later primitives stay designed-not-built. The seam is built so `Skill` is the *first* concrete input, not the only possible one: `Analyzer<Input, A>` / `Evaluator<Input, A>` take a generic input ([¬ß2](#2-glossary--the-domain-language), [¬ß3.1](#31-the-skill-analysis-seam-the-spine)). Broadening the product means adding more **equipment primitives** ‚Äî declarative artifacts that change what an agent can do, know, decide, or safely access ‚Äî each with its own source model, analyzer/lint, renderer, and optionally an evaluator, all on the existing seam (no new pipeline).

The broadening earns its keep only when a **second** primitive ships and **composition** becomes possible: cross-primitive evaluation questions the single-primitive product can't ask ‚Äî

- Does this Skill call this Tool correctly?
- Does the Tool's output match the Response schema?
- Does the Skill stay inside the declared Policy?

Composition is **additive on the seam** ‚Äî an evaluator gets richer input context, it does not become a new pipeline.

**Primitive order (each earns the next):**

1. **Response schemas** *(built, [#130](https://github.com/CrowBe/agentbranch/issues/130); chat-buildable, [#151](https://github.com/CrowBe/agentbranch/issues/151))* ‚Äî structured output definitions (JSON Schema documents). The `response-schema` module carries the lossless source model and a pure, zero-token lint (structural validity, required fields, smell rules) rendering through the seam renderers, plus the offline schema-subset validator and example-value generator the compositions below use. Authoring rides the build loop's shape: one frozen cacheable system prompt (interview-first, anchored on one real filled-in example of the output) with a `write_response_schema`/`edit_response_schema` tool pair through the model gateway, surfaced in the workspace's Equipment mode; the finished document is quality-checked and session-kept exactly like a pasted one. Each later primitive gets the same treatment ‚Äî its own frozen prompt + tool pair ‚Äî as it becomes chat-buildable.
2. **Tool contracts** *(built, [#131](https://github.com/CrowBe/agentbranch/issues/131); backend authoring path, [#175](https://github.com/CrowBe/agentbranch/issues/175))* ‚Äî typed input/output plus descriptions, examples, failure modes, safety notes. The `tool-contract` module mirrors the same shape (lossless source model + pure lint, inline schemas run the shared schema shape rules); its I/O references response schemas via `$ref` by title. Authoring rides the same build-loop shape as response schemas: one frozen cacheable prompt anchored on the tool's job, example input/output, failure modes, and confirmation boundary, with a `write_tool_contract`/`edit_tool_contract` pair through the model gateway and lint feedback after writes. The first relational evaluation is also built ([#132](https://github.com/CrowBe/agentbranch/issues/132)): **Test run** takes a **bundle**, the mock-tool registry is driven by the contracts (mock output conforms to the output schema), and each observed call is validated ‚Äî arguments against the input schema, handled output against the output schema ‚Äî surfacing in Insights, the breakdown, and eval feedback.
3. **Policies / guardrails** ‚Äî declarative constraints on allowed actions, confirmations, and data/network/file/output access. Lint catches obvious violations; evaluation tests whether the constraint holds under pressure.
4. **Agent profiles** (later) ‚Äî durable identity + defaults (system prompt, role, tone, model preference, default equipment). Analysis is easy; the evaluation half needs rubric / LLM-judge support.
5. **Knowledge packs** (later) ‚Äî bounded reference knowledge, lightweight RAG-adjacent, no vector DB. Meaningful only when evaluated as part of a profile + skill + tool bundle.

Supporting work on the way: **user-authored test materials** (let users seed Triggering eval / Test run with their own positive/negative prompts and scenarios ‚Äî extends existing evaluators before becoming a top-level primitive) and **regression comparison** across evaluation records ([¬ß6](#6-data-model-sketch); Richer evals above) so a revision's improvement or regression is visible.

**Smallest useful composition** *(built ‚Äî the bundle test run above)* ‚Äî the bar for being more than a single-primitive tool: *here is a Skill, a Tool contract, and a Response schema; does the Skill call the Tool correctly and produce valid output?* Broader than single-primitive validation without dragging in container runtime, vector DBs, or subagent routing ‚Äî the **non-goals for the first expansion**.

### 9.3 Branching iteration ‚Äî draft, main version, promote

**This is built** ‚Äî data substrate ([#128](https://github.com/CrowBe/agentbranch/issues/128)) and user-facing surface ([#129](https://github.com/CrowBe/agentbranch/issues/129)). The schema, the branch/promote repository methods, and the daily retention job ([¬ß6](#6-data-model-sketch)) live behind the `SkillRepository` + `SkillRetentionRepository` ports; the surface wires them into build + evaluate: a draft build appends to the branch head (`saveToBranch`, main pointer untouched), evaluation records pin the draft head (a shared version resolver over `branchId`, [¬ß3.1](#31-the-skill-analysis-seam-the-spine)), and the hero carries the draft-vs-main legibility banner with the **Start a draft** / **Set as main version** / **Discard draft** controls plus a resume affordance for open drafts. Versioning is **draft / main / promote** so a user can iterate and evaluate without disturbing the version they trust.

**The model.** A skill has one **main version** ‚Äî the blessed pin everything defaults to (export, install, the hero's first paint). Iteration happens on a **draft**: a lineage of revisions that accumulates *without moving the main pointer*. The user evaluates the draft (lint, test run, triggering eval all run against the draft's head, not main), and when satisfied **sets it as the main version** ‚Äî promote moves the pin to the draft's head. A skill may carry **several open drafts at once**: a new build session forks a fresh draft off main or resumes an open one, so an un-promoted draft from a prior session is never stranded and never silently becomes main. This is what made multiple drafts the call over a single draft slot ‚Äî a one-draft model forces a stale/resume/discard decision the moment a second session starts; coexisting drafts remove that state machine.

**Git shape, stated precisely.** Internally, *main is just a branch* ‚Äî every version belongs to a branch, one branch is flagged the blessed/main lineage via an explicit pointer, and promote re-points it. Versions chain by **parent pointer** (a DAG), so two drafts advancing independently never collide ‚Äî this is why the linear monotonic `revision` demotes to a per-draft display ordinal ([¬ß6](#6-data-model-sketch)). **Promote is replace-not-merge:** setting a draft as main repoints the pin to that draft's head; it is *last-promote-wins* with no three-way merge. Instruction-only skills carry no merge-conflict semantics worth solving, so a draft based on a now-superseded main stays a valid draft the user can re-promote (overwriting) or discard. **No merge, no rebase** ‚Äî the accepted simplification that keeps the SMB model a single verb.

**Draft ‚âÝ session.** The draft is the durable artifact; a build session is ephemeral. A session records which draft it last touched (for resume), but a draft outlives its session ‚Äî there is no session entity ([¬ß6](#6-data-model-sketch)), so resume is a pointer, not a lifecycle.

**Copy contract ‚Äî git shape, not git vocabulary.** User-facing nouns and verbs: a branch is a **draft**; the blessed version is **the main version**; promote is the button **"Set as main version"** ("main" is *semantic* ‚Äî the main one ‚Äî not git's branch name). The internal/code terms `branch` and `promote` never reach user copy, the same internal-vs-surface split as mock-tool registry / test run and tap / Publish ([¬ß9.1](#91-skill-tap--open-distribution-with-safety-badges)). **"Publish" is reserved for the tap** ‚Äî where a skill genuinely becomes visible to others ‚Äî and must not be reused for promote, which changes nothing's visibility.

**Retention is a daily tidy, not an inline cap.** Pruning is off the write path (no prune inside a save/promote transaction): it runs as a **fixed-cadence cleanup job** ‚Äî the `SkillRetentionRepository` port, driven by a **Vercel Cron** trigger (`/api/cron/retention`, locked behind `CRON_SECRET`) ‚Äî which the UI can state plainly ("older drafts are tidied up daily; we keep your last N per draft and your main version"). The job enforces two axes, neither during a live session: **per-draft history depth** (latest-N-per-draft) and a structural **open-drafts-per-skill cap** (one tunable constant, the same bounded-by-construction shape as the free-tier skill-count cap, [¬ß8](#8-free-tier)). It **never prunes the main lineage or any open draft's tip** ‚Äî only interior history beyond N. Moving retention off the write path also dissolves the "don't prune an in-flight draft out from under the user" hazard *by construction*: nothing is pruned while a session is live. Schema delta + the retention port live in [¬ß6](#6-data-model-sketch).

**On the seam.** Branching is **upstream of the skill-analysis seam**, not a capability on it: it changes *which `SkillSource`* feeds an analyzer/evaluator, never the seam's `artifact ‚Üí render` shape. No new `ArtifactKind`, no renderer change ‚Äî every capability (hero, lint, visualise, export, test run, triggering eval) runs against a draft head exactly as it runs against main today. The substrate is therefore a **port + adapter** change (MODULE_DESIGN ¬ß6 rule 2) on the existing `SkillRepository` plus one new retention port ‚Äî not a new pipeline ‚Äî implemented in both the Prisma and memory adapters and tested as one contract.
