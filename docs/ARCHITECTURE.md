# agent.branch — Architecture

Single source of truth for **what we build and why** — product, system, services, data, **and** app/screen layout. All of it lives here because it's one *kind* of knowledge (decisions that translate into code), regardless of layer.

**Companion docs:**

- **`DESIGN.md`** — the *visual design system* (theme, color tokens, type scale, spacing, components).

---

## 1. Product thesis

**agent.branch is a skill *testing / CI* tool that also authors.** That is the long-term thesis, and it's the sharp end of the pitch — most tools stop at editing; the durable value is *validating* a skill before it ships.

In the near term it's a **visual authoring + lightweight-validation tool for agent skills**: build a skill in chat with a live-streaming preview, **visualise** its logic, **test-run** it, check its **triggering**, and **export** it. The frame is **Claude-first runtime, standard-native artifact**: authoring and validation run on Claude, and the skill the user owns is an [Agent Skills open-standard](https://agentskills.io) `SKILL.md` folder that installs as-is across compatible tools (Claude, Codex CLI, Gemini CLI, Copilot and more). **Honest portability** is the growth edge — validating *behaviour* on other runtimes' models, never a "runs everywhere" fidelity claim; the standard already solved the format question.

**Audience is a bridge, not just developers.** Two users:

- the **technical builder** — fluent in `SKILL.md`, YAML, trigger logic;
- the **non-technical SMB owner** — automating admin (inbox, scheduling, docs) with AI.

A skill is an inherently technical artifact (instructions + frontmatter + trigger logic), so the product's job is to make it *approachable without dumbing it down*: **one warm-pro identity, technical depth on demand, not in your face.** This is a positioning commitment, not just styling — it drives the hero's [Rendered/Source render model](#7-frontend--app-shell) and the whole `DESIGN.md`.

**The wedge is the hosted graphical flow.** The eval tooling that exists elsewhere is CLI-bound (Anthropic's skill-creator runs trigger evals as a terminal meta-skill), so for the builder who will never run a CLI, a hosted surface — evaluations launched from the hero's chips, landing as Insights — is the only validation on offer. That gap, plus **honest cross-runtime results** ([#116](https://github.com/CrowBe/agentbranch/issues/116)) and **honestly-labelled distribution** ([§9.1](#91-skill-tap--open-distribution-with-safety-badges)), is the defensible ground; a triggering eval alone is table stakes the CLI tools already provide. **Teams** (shared libraries, private validation gates, seats) are the named later revenue layer, not a v1 audience.

**v1 = a thin vertical slice across all capabilities, Claude-only.** The authoring loop ships polished; everything else ships thin. Full breadth, minimum depth.

## 2. Glossary — the domain language

One term per concept. Use these names everywhere — in docs, code, and UI copy.

| Term | Definition |
|---|---|
| **Skill** | The product's unit of work: a reusable instruction set for an AI agent, per the **Agent Skills open standard**. Instruction-only — no bundled runnable code. Claude is the first-class authoring and validation runtime; the artifact itself is standard-native and installs across compatible tools. |
| **`SKILL.md`** | The skill's source file (the open standard's format): YAML frontmatter (`name`, `description`) + markdown body (instructions, workflow, rules). |
| **Skill record** | The persisted skill in our DB (see [§6](#6-data-model-sketch)). Exports are rendered *from* it. |
| **Skill metadata** | A skill's discovery metadata: one **category** from a closed taxonomy + up to 8 kebab-case **tags**, living in `SKILL.md` frontmatter (so it travels with the standard-native artifact and is pinned by the content hash). Drives Skill-library filtering/search and the profile pages' "class" grouping. Authored as a mix — build-loop drafting writes it, the author edits it, the **metadata-suggest** capability recommends editable name, description, category, and tags through the local → gateway → deterministic ladder. |
| **Build loop** | The core agentic loop: Claude writes/edits the `SKILL.md` through the `write_skill`/`edit_skill` tools, streaming to the preview. Reaches the model only through the **model gateway** (`streamAgent`), so a build turn is gated + accounted like every other model call. The loop is *closeable* with evaluation and lint results: an **eval feedback** message can be appended to the conversation, giving Claude the specific failure cases and rationale it needs to make targeted revisions. |
| **Eval feedback** | A formatted summary of an **Evaluation result** (or **Lint** artifact) injected as a user message into the build loop conversation, so Claude can revise the skill against observed evidence rather than guessing. Produced by a **feedback formatter** — a pure function in the `build-loop` module that translates the structured artifact into actionable revision context. User-triggered from the **Insights** surface for evaluation results ("Revise with this feedback"); auto-injected after `write_skill` for lint findings. Distinct from **Insights** (the user-facing display surface) — Insights is what the user reads; eval feedback is what Claude reads to author the revision. |
| **Skill-analysis seam** | The architectural spine. The shared pattern **read input → emit a structured artifact → render it for a surface**. Built once; each feature is a new capability on it, not a new pipeline. Carries **two capability shapes** (below). Distinct from the skill IR: the seam is the pattern, the IR is one artifact type on it. The `Analyzer<Input, A>` / `Evaluator<Input, A>` input slot is generic: `Skill` was the first concrete input, and the **equipment primitives** (response schemas, tool contracts, [§9.2](#92-equipment-primitives--composition-broadening-beyond-skill)) ride the same seam. |
| **Analysis capability** | A *static* capability on the seam — derives a structured view from an input's source. Pure, runs offline. Wraps an `Analyzer<Input, A>` (`analyze(input)`). The Rendered/Source hero, Visualise, Lint and Export are analysis capabilities over `Skill`; Response schema quality and Tool contract quality are analysis capabilities over their own primitives. |
| **Evaluation capability** | A *dynamic* capability on the seam — **runs an input through a model** and observes behaviour. Costs tokens, needs a model (fails `model_unavailable` offline). Wraps an `Evaluator<Input, A>` (`evaluate(input, gateway)`) that **owns its method, not its resources**: it builds its own conditions (Scenario / distractors / battery) but model access is handed in via the **model gateway**. Triggering eval is an evaluation capability over `Skill`; Test run's input is a **bundle** — the Skill, optionally with Tool contracts + Response schemas. |
| **Evaluation result** | The artifact an evaluation capability emits — the structured run-record. Ephemeral; **never shown raw**. Renders to **Insights** (and a detailed breakdown on demand). Distinct from the persisted **evaluation record** ([§6](#6-data-model-sketch)): the result is rendered now, the record is stored and re-rendered later. |
| **Insights** | The default, plain-language rendered surface of an evaluation result — meaning the user can act on, not a data wall. The audience bridge ([§1](#1-product-thesis)) lives in the *renderer*; a detailed breakdown sits behind it for depth (same Rendered/Source duality as the hero). |
| **Model gateway** | The platform's **single, controlled, metered entry to the model** — its own module. Owns the AI-SDK call plumbing; exposes fine intent-level primitives `classify`/`runAgent`/`streamAgent`/`generate` that callers compose into their own method. Pure mechanism — knows no capability kinds. It does **not** pick the provider/model or hold the key: it resolves a `LanguageModel` per call from the **model router**. Every call carries an **accounting tag** (`account` \| `platform`); the gateway routes accounting through the **usage** authority. Consumers: the build loop (via `streamAgent`) and evaluation. Cross-runtime validation uses its per-call target selection; mock-data generation also routes through it. **Target state ([#34](https://github.com/CrowBe/agentbranch/issues/34), [#35](https://github.com/CrowBe/agentbranch/issues/35)):** as the one metered chokepoint, the gateway is also where input-size budgets, prompt caching (a frozen system prefix), and cache-aware token accounting are enforced and measured. |
| **Model router** | The platform's **single provider + model selection authority** — its own module, the layer beneath the gateway. Owns the **provider registry** (every provider the platform knows: Anthropic default, Nous Portal, extensible), their **credentials** (a server-pool key per provider, plus an optional **bring-your-own override** that wins when present), and the runtime-mutable **active selection** (which provider + model each primitive routes to). The gateway calls `resolve(primitive)` to get a `LanguageModel`; the **model console** drives the router's mutators so provider/model can be switched (or a key connected) at runtime, e.g. to rotate providers while testing the flow. Pure *selection* mechanism — knows no accounting or capability kinds. Per-capability routing (Opus for `streamAgent`, cheaper models for `classify`/`generate`) is expressed here as per-primitive model ids. Selection + bring-your-own keys are **process-local and single (not per-user)** in v1 — one shared active selection per instance, which suits the testing/rotation use case; per-user isolation + a persisted store is a future port + adapter swap. Keys never leave through its secret-free snapshot and are never logged. |
| **Local suggestion provider** | The browser-side progressive-enhancement option for light, editable suggestions. It sits beside — never inside — the model gateway/router because it spends no platform tokens and skill content stays on the user's device. It may propose text the author sees and edits; it never writes an artifact, gates an action, or produces an **Insight**. The workspace uses it only when the browser reports a model already **available** (never downloadable), and silently falls through to the equivalent server route otherwise. User copy says **suggested on your device** only when this rung served the result. |
| **Accounting tag** | Declared by the *caller* on every model-gateway call: **`account`** (user-attributable, spends the user's free quota) or **`platform`** (the platform's own cost to enable a feature — e.g. generating mock data — never charged to a user's quota). Two accounting streams: account = quota admission + priced token recording ([§8](#8-free-quota)); platform = our own cost ledger (deferred). A model call denied by a cap fails `cap_reached` (distinct from `model_unavailable` = no model at all). |
| **Skill IR** | *One specific* artifact type on the seam: visualise's intermediate representation — nodes + edges, each carrying a **source-span** back into `SKILL.md`. A visualise concept, not the whole seam. |
| **Test run** | User-facing term for executing a skill against **mocked** tools to see how it behaves. The mechanism is the **mock-tool registry**; the agent tool is **`execute_skill`**. Nothing real is ever touched. (Always "test run" in user copy — never "sandbox": jargon that intimidates rather than informs.) |
| **Mock-tool registry** | The mechanism behind a test run: when the skill calls a tool (e.g. "fetch unread email"), the registry returns generated mock data instead. The skill runs end-to-end against fake tools, so the user sees its behaviour without anything real happening. No code execution or containers — skills are instruction-only, so there's nothing to containerise. |
| **Response schema** | The first equipment primitive beyond Skill ([§9.2](#92-equipment-primitives--composition-broadening-beyond-skill)): a structured output definition — a JSON Schema document — with a lossless source model and pure offline lint. No evaluator; its schema subset also validates tool-contract examples and test-run calls. Chat-buildable through the build loop's shape under its own frozen authoring prompt ([§9.2](#92-equipment-primitives--composition-broadening-beyond-skill)). |
| **Tool contract** | The second equipment primitive: a tool's typed input/output plus description, examples, failure modes, and safety notes. Its I/O is an inline schema or a `$ref` to a **response schema** by title. Standalone it is an analysis primitive (lossless source model + lint) and chat-buildable through the build loop's shape under its own frozen authoring prompt; in a **bundle** it drives the test run's mock tools and per-call validation. |
| **Bundle** | The test run's composed input ([§9.2](#92-equipment-primitives--composition-broadening-beyond-skill) *smallest useful composition*): the Skill plus selected Tool contracts and the Response schemas they reference. Contracts drive the mock-tool registry (mock output conforms to the contract's output schema) and each observed call is validated — arguments against the input schema, handled output against the output schema. |
| **Triggering eval** | The v1 validation: does the skill *fire* on the right prompts and *stay silent* on the wrong ones? Run competitively against a **distractor library** + a positive/negative **prompt battery**. |
| **Safety rating** | *User-facing term* for the persisted result of a **safety review** run ([§9.1](#91-skill-tap--open-distribution-with-safety-badges)): verdict (passed / needs review / blocked) + per-class risk scores + Insight, pinned to the skill version reviewed. **Opt-in and manual, never automatic** — offered any time for an unrated version and as an optional, non-gating step of "Set as main version". A passing rating pinned to a published version's hash earns the **safety badge**; a published version without one is labelled *potentially unsafe — not validated*. Runs spend `account` (`safety-review` capability); the capability stays caller-tagged. |
| **Cross-runtime validation** | The portability surface. Skills travel as-is under the open standard, so portability is a *behaviour* question, not a format one: one engine runs the skill's triggering battery against selected runtime targets through the model gateway and reports a per-target grid. Results are honest, model-level approximations of each tool's runtime — copy says so. |
| **Rendered / Source view** | The hero's two views of the same skill. **Rendered** (default) = friendly sans-serif document. **Source** = raw `SKILL.md` monospace. Both are renderers on the seam. |

## 3. Core architecture — the through-line

A single **server-side agentic harness** (Vercel AI SDK + Claude) with a **tool registry**. Not N services — one agent loop, many tools.

```
Browser (React)  ──SSE──▶  Next.js route handler
       ▲                          │
       │ stream tool output       ▼
   preview / viz / chat     Build loop  ──streamAgent──▶  Model gateway ──resolve──▶ Model router
                                  │                      (metering + accounting)    (registry + creds
                                  ▼                              │                   + active selection)
                            Tool registry:                       ▼                          │
                              write_skill, edit_skill,     Claude / Nous (Vercel AI SDK) ◀──┘
                              visualise_skill, execute_skill,                  ▲
                              + mock-tool registry integrations          Model console (UI)
```

- The **model router owns the provider keys** + the active provider/model selection; the **gateway** owns metering + accounting. Nothing above the gateway (the build loop, the route handler) touches the raw model, the keys, or the client. The **model console** is the only surface that mutates the selection, through the gateway-adjacent router.
- Tool-call output streams to the client over **SSE**. Evaluation runs use the same transport for progress events and the final artifact.
- `execute_skill` runs against the **mock-tool registry** ([§2](#2-glossary--the-domain-language)): a test run, not real execution.

### 3.1 The skill-analysis seam (the spine)

Several capabilities are the **same shape**: *read an input → emit a structured artifact → render it for a surface*. Build the seam once; each feature plugs in without a new pipeline. The generic `Input` slot is load-bearing: most capabilities read a `Skill`, the equipment primitives (response schemas, tool contracts, [§9.2](#92-equipment-primitives--composition-broadening-beyond-skill)) read their own source models, and the test run reads a **bundle**.

The seam carries **two capability shapes** — same `artifact → render` tail, different head:

- **Analysis** (static) — derives an artifact from the input's source. Pure, runs offline. Wraps an `Analyzer<Input, A>` (`analyze(input)`). Its artifact is *recomputed on demand*, never persisted. Rendered hero, Source view, Visualise, Lint and Export use `Skill` as their input; Response schema quality and Tool contract quality use their primitive's source model.
- **Evaluation** (dynamic) — produces an artifact by **running the input through a model**. Needs a model (fails `model_unavailable` offline), costs tokens. Wraps an `Evaluator<Input, A>` (`evaluate(input, gateway)`) that **owns its method, not its resources** — it builds its own conditions but is handed the **model gateway**, the platform's single metered entry to the model. It composes its method from the gateway's fine primitives (`classify`/`runAgent`); it never touches the raw model, the key, or token accounting. Its artifact (an **evaluation result**) is *persisted* as an evaluation record and renders to **Insights**. Triggering eval uses `Skill` as its input; Test run uses a **bundle** (the Skill, optionally with Tool contracts + Response schemas).

> The split is named so a new capability is classified **before** it's built: "is this analysis or evaluation?" decides whether it gets an `Analyzer` or an `Evaluator`, runs offline or needs the harness, recomputes or persists. The seam stays one spine; the two heads keep the model dependency and token cost out of the pure analysis surfaces.

| Capability | Shape | Extract / run | Render (v1) | Render (later) |
|---|---|---|---|---|
| **Rendered hero** | analysis | frontmatter + body | friendly structured document (sans-serif) | richer doc layout / inline editing |
| **Visualise** | analysis | skill IR (nodes+edges+source-spans) | Mermaid | interactive canvas (React Flow) |
| **Lint** | analysis | frontmatter + body quality rules → `LintReport` | Insights (score/grade + actionable findings) + Breakdown (full finding list with source-spans) | per-primitive rule sets as new equipment types land |
| **Response schema quality** | analysis | JSON Schema document structure + smell rules → LintReport-shaped artifact | Insights + Breakdown (same quality vocabulary as Lint) | referenced-schema graph checks |
| **Tool contract quality** | analysis | I/O typing completeness, description/example quality, failure modes, safety notes → LintReport-shaped artifact | Insights + Breakdown | resolve `$ref`s against a schema library |
| **Cross-runtime validation** | evaluation | run triggering battery on selected runtime targets → per-target results | Insights | per-target results grid |
| **Test run** | evaluation | run the bundle vs. mock-tool registry (contract-driven when contracts are supplied) → result + per-call contract checks | **Insights** (+ transcript & contract checks on demand) | richer scenarios / multi-run |
| **Triggering eval** | evaluation | run skill vs. distractors + battery → result | **Insights** (+ pass/fail breakdown) | scored / workflow / regression |

This is what "build it flexible" means here: a **clean seam**, not a speculative v2 abstraction.

## 4. Decisions

Current choices and the reasoning behind them. Live — open to revision while we're still in flux.

| Branch | Decision | Why |
|---|---|---|
| Build loop | Claude on our key (Anthropic) via **Vercel AI SDK**, reached through the **model gateway** (`streamAgent`) | Skill primitive is Claude-tuned; routing through the gateway means build turns are gated + metered by the one accounting authority, like every other model call; one harness, provider-swappable at runtime via the router |
| Build loop — initial flow | **Requirements interview before the first draft** — on a new skill the authoring agent interviews the user (small numbered rounds, plain language, a readiness checklist that gates `write_skill`) instead of drafting immediately; it presses on scope and recommends splitting bundled jobs into companion building-block skills ("building blocks that work together", never "composable" in user-facing copy) | A first draft from a thin request bakes in guesses that anchor every revision; pressing on requirements and right-sizing up front is where authoring quality is won. Bounded rounds + skip conditions (a fully specified request, or "just draft it" — an explicit command the interaction panel's empty state hints verbatim and the agent mentions once in its first round) keep it an interview, not an intake form — preserving the bridge audience ([§1](#1-product-thesis)) |
| Eval → build feedback | Evaluation results feed back into the build loop as **eval feedback** messages — a formatter in the `build-loop` module (not the eval modules) translates the structured artifact into a user message; the client appends it and re-submits | No new API, no new gateway primitives; the formatter lives in `build-loop` because "what does Claude need to revise this skill?" is a build-loop concern, not an evaluation concern; the high-value content for triggering eval is the failed cases with model rationale (the classifier's own explanation), not just the Insight headline |
| Provider/model routing | A standalone **model router** beneath the gateway owns the provider registry, credentials (server pool + bring-your-own override), and the runtime-mutable active selection; the **model console** switches it live | Splitting *selection/credentials* (router) from *metering* (gateway) keeps each pure; runtime switching lets us rotate providers/models to test the flow without redeploying, and bring-your-own keys layer on without breaking the server-owns-key default. Default selection still comes from env, so nothing changes for a plain boot |
| Local inference boundary | On-device inference is a **local suggestion provider** beside the model gateway, limited to editable suggestions. It is not a model-router provider: it runs in the browser on the other side of the HTTP/SSE boundary, has no metered accounting, and cannot produce Insights or write artifacts. Progressive enhancement keeps the UI identical elsewhere: a session-cached availability probe accepts only `available`; `downloadable`, malformed output, and failures silently use the existing route. Prompt API sessions are per suggestion, structured with a JSON-schema response constraint, and long skill source is truncated at the provider boundary. | The gateway remains the single metered model entry while cheap local suggestions keep skill content on-device and never prompt an SMB owner into a multi-gigabyte model download. Per-suggestion sessions avoid retaining skill content or scarce browser model context between actions; the route fallback preserves feature parity. |
| Model console access | The console + its route are **admin-gated** (an env allowlist of Clerk user ids / emails, checked by `isAdmin`): open on a no-auth dev box, admin-only when auth is configured, locked when auth is on but no allowlist is set | The selection is **instance-wide** in v1 (one shared active provider/model + keys), so a plain signed-in user must not change what everyone runs on. Fail-safe (deny without an allowlist) beats fail-open for a credential surface; per-user config is the heavier future path |
| Skill scope | **Instruction-only** (`SKILL.md` + reference docs); no bundled runnable code | A test run becomes a mock-tool registry, not a container — removes all code-exec/isolation infra |
| Skill metadata | **`category` + `tags` live in `SKILL.md` frontmatter** — one category from a closed taxonomy (the `skill` module's `SKILL_CATEGORIES`) plus up to 8 kebab-case tags. The **metadata-suggest** surface proposes editable name, description, category, and tags through three first-available rungs: local provider, metered gateway route, then that route's deterministic fallback. Every rung has the same UI and suggestions write only after author acceptance; acceptance checkpoints the saved skill's active main or draft lineage, while an unsaved workspace is labelled as such. A changed description nudges Triggering eval. Lint nudges, never gates: missing category = info, unknown category = warn | Frontmatter is the one home, so metadata travels with the standard-native artifact and is pinned by the same content hash the safety badge trusts. The ladder spends nothing when local is available without creating a capability gap elsewhere; explicit acceptance protects the triggering surface from silent rewrites. |
| Write / edit | `write_skill({content})` streams the whole doc on first draft; `edit_skill({old,new})` applies a highlighted diff on revisions | Mirrors Claude Code's Write+Edit; cheaper tokens; preview is a doc model supporting replace+patch |
| Visualise | Model emits a **skill IR** (nodes+edges, each with a **source-span** into `SKILL.md`); v1 thin **IR→Mermaid** renderer; diagram type model-chosen; 1 generation | IR is the stable contract; the later canvas reuses it; node↔source mapping paid once |
| Test run — mock data | Fixed schema per integration; **contents generated to stress the skill**; cached per session for deterministic re-runs | Relevant to the user's real case; billable token cost; deterministic within a session |
| Test run — tool selection | Mock tools **auto-inferred** from the skill; user can override | One-click test; manual add for undetected tools |
| Triggering eval | **Lightweight** — does the skill fire on the right prompts, stay silent on the wrong ones? Mirrors Anthropic's skill-builder output | Cheapest eval, no judge model, tests the #1 failure mode (bad description/triggers) |
| Triggering eval — competition | User skill + **distractor library** (~20 → ~30) + positive **and** negative **prompt battery** | Skill selection is competitive; testing in isolation gives false confidence |
| Cross-runtime validation | **Honest behavioural check** — run the skill's triggering battery against other runtimes' models via provider swap through the model gateway; *not* format conversion, *not* "skills run everywhere" | The open standard makes the file portable everywhere; the open question is behaviour — "will my skill survive Codex/Gemini?" — and honesty is the better pitch |
| Export v1 | Copy button + **`.zip`** of the standard skill folder (`skillname/SKILL.md` + refs) | The standard folder is the one installable artifact across compatible runtimes — no per-target packages needed (every major tool consumes `SKILL.md` natively); copy serves the paste-it case |
| Authoring screen layout | **Preview-primary**: the streaming skill document is the hero; interaction is a **demoted control surface**, not a co-equal chat panel | The skill artifact is the product, not the conversation |
| Hero render mode | Hero has **two views: Rendered (default) + Source (toggle)** — see [§2](#2-glossary--the-domain-language). Default Rendered for newcomers; Source one click away | Bridge audience ([§1](#1-product-thesis)): raw monospace `SKILL.md` is the biggest "this is for programmers" signal. Rendered-default removes that wall without hiding the artifact. **Reuses the seam — Rendered is just another renderer**, so it's cheap |
| App shell | **Thin branded top bar** (chrome only, no nav links) + **left slideout menu** (all primary nav + account footer; collapsed icon rail by default, expands on demand) wrapping the hero + slim right interaction panel. Tool surfaces are **chips** on the hero header | Top bar stays clean framing; nav consolidated in the menu; hero gets max width by default |
| Stack | Next.js (App Router) + TS · Postgres via **Prisma 7** (pg driver adapter) · **Clerk** auth (Google+GitHub) · Vercel AI SDK (Claude default, optional Nous Portal) · SSE · Vercel + Neon/Supabase | One language end-to-end; fastest empty-repo-to-deployed; the build loop streams from a route handler, reaching the model through the gateway that holds the key |

## 5. v1 thin-slice spec (per capability)

The build loop is **not thin** — it's the core and must feel good. Everything else is the minimum that proves the shape.

1. **Auth + meter** — Clerk (Google+GitHub), Postgres, per-turn + per-token counter. Thin = no admin UI, enforced caps only.
2. **Build loop** — Claude via the model gateway's `streamAgent`; `write_skill`/`edit_skill`; SSE to preview. **Polished, not thin.** Opens with the **requirements interview** ([§4](#4-decisions)) before the first draft. Closeable with **eval feedback** — triggering-eval and test-run Insights feed back as messages, giving Claude the failure cases and rationale to make targeted revisions.
3. **Lint** — pure analysis, zero tokens, uncapped; auto-runs on every version after `write_skill`. Insights + Breakdown renderers. The seam's `Analyzer<Input, A>` design makes lint rule sets portable to future equipment primitive types.
4. **Import** — paste `SKILL.md` text or import a public GitHub skill URL; same parse → persist → render path as authored skills.
5. **Visualise** — model emits skill IR; thin IR→Mermaid renderer; 1 generation; no diagram editing.
6. **Test run** — single run, 1 generated scenario, mock-tool registry with 1–2 mock integrations (email first), Claude only.
7. **Triggering eval** — corpus-fed distractor library (~20), small prompt battery, "did it fire?" boolean. No judge model yet.
8. **Export** — copy + the standard skill folder `.zip`.
9. **Cross-runtime validation** — validation-first engine: run the triggering battery against selected runtime targets through the model gateway, with unconfigured targets shown honestly.
10. **Billing** — the free initial quota on the token + cost meter; every capability open while quota lasts. Thin = no paid top-ups, no PAYG.

## 6. Data model (sketch)

- **`users`** — identity from Clerk (Google/GitHub).
- **`skills`** — `id, user_id, name, description, body, frontmatter_json, main_version_id, created_at`. `main_version_id` is the **explicit blessed pointer** — the version everything defaults to (export, install, the hero's first paint). On the linear path it tracks the newest revision; branching makes it explicit and independent of newest, so a draft can append higher revisions without becoming main. *Main is just the branch that owns this version.*
- **`skill_branches`** — a line of iteration (a **draft** in user copy, [§9.3](#93-branching-iteration--draft-main-version-promote)): `id, skill_id, status (open | discarded), ordinal`. Every version belongs to one; the branch owning the skill's `main_version_id` *is* the blessed lineage, so `isMain` is derived, never stored. A skill carries several open drafts at once.
- **`skill_versions`** — append-only revisions of a skill (export is a pure function of a version; enables future regression evals). Versioning is **draft / main / promote** ([§9.3](#93-branching-iteration--draft-main-version-promote)): a version belongs to a branch and chains by `parent_id` (a DAG), so two drafts advancing independently never collide. `revision` is a **per-branch display ordinal** (`@@unique([branch_id, revision])`), not a per-skill monotonic counter. **Restore** lands a copy as a new head revision and **promote** re-points `main_version_id` to a draft's head — both append-only (no version is mutated or deleted). Retention is **off the write path** — a **daily cleanup job** (the `SkillRetentionRepository` port + a Vercel Cron trigger, [§9.3](#93-branching-iteration--draft-main-version-promote)) enforces latest-N-per-draft depth *and* an **open-drafts-per-skill cap**, never pruning the main lineage or an open draft's tip. Pruned draft versions release their pins via `onDelete: SetNull`, so stored run records survive.
- **`usage`** — per-user token + turn + used/reserved cost counters plus append-only price-keyed charge records (micro-USD; the buckets drive paid metering later).
- **`publications`** — public addresses for skill versions ([§9.1](#91-skill-tap--open-distribution-with-safety-badges)): publisher, `owner/skill-name` slug, trust tier (private / published / reviewed), pinned append-only `skill_version_id`, and content hash. Publishing is ungated; the version's safety rating (or its absence) drives the badge/flag presentation, and a changed version means a different hash — unbadged until re-analysed.
- **`harness_versions`** — append-only identity for the validation harness itself: one manifest hash decomposed into sub-hashes for the build-loop system prompt, lint ruleset, prompt-battery generator, test-run world generator, distractor library, plus the deploy git SHA when available. Source artifacts stay in git; the row records identity + hash so outcomes are attributable to the harness that produced them.
- **`eval_runs`** / **`test_runs`** — recorded triggering-eval and test-run results. Each run carries nullable `skill_version_id` and `harness_version_id` pins: the skill revision evaluated and the harness manifest that produced the result. Both use `onDelete: SetNull`, so stored runs survive version cleanup or harness-retention changes. List/fetch APIs plus My skills / History surfaces re-render stored records through the seam renderers.
- **`safety_ratings`** — recorded **safety ratings** ([§9.1](#91-skill-tap--open-distribution-with-safety-badges)): verdict + per-class scores + Insight for one safety-review run, with the same nullable `skill_version_id` / `harness_version_id` pins (`onDelete: SetNull`) as the other evaluation records. The version pin is load-bearing twice over: "does this version carry a rating?" is a `latestForVersion` read, so an unrated head is what triggers the opt-in offer — and the same read is what decides whether a published version renders the safety badge or the potentially-unsafe label ([§9.1](#91-skill-tap--open-distribution-with-safety-badges)).
- **`benchmark_runs`** — scorings of the frozen regression-benchmark set, each pinned to the harness version in effect and the set hash it scored ([§9 *Harness improvement loop*](#harness-improvement-loop-admin)). Append-only; admin-only surface, so no user scoping.

**Persistence invariants.** All DB access goes through Prisma's query builder — **no raw SQL anywhere**, so stored skill content can never become a SQL payload (parameterized-only). ESLint bans raw SQL calls unless a reviewed exception is added. Row ownership is enforced *in the query* (`where: { id, userId }`) for user-owned records rather than only re-checked in callers. **Target state ([#34](https://github.com/CrowBe/agentbranch/issues/34)):** `frontmatter_json` bounded (size / depth / key-count) with unsafe keys (`__proto__`, `constructor`) rejected on parse. Skill content and prompts are never written to logs.

Export shapes are **rendered from** the skill record — the schema is export-agnostic.

Presentation-layer architecture — same kind of decision as the rest of this doc (it determines components, shell, routing), so it lives here.

First-run needs a one-time hint so the expandability is discoverable.

## 8. Free quota

The free tier is one number: a **free initial quota** — **$1.00** of model spend granted per account at sign-up (`INITIAL_QUOTA_MICROS`, one tunable constant). Spend admission is capability-blind, so **every capability is open while it lasts**, triggering evals and safety ratings included. Before a call opens, the usage repository atomically reserves its conservative maximum cost (input bytes as the input-token upper bound plus the primitive's output ceiling); concurrent calls therefore cannot oversubscribe the grant. Completion reconciles the hold to actual token buckets at the resolved model family's versioned micro-USD price, recorded in an append-only charge row; provider failures release it. Unknown models fail closed until priced. The quota never resets. The top-bar dollar balance refreshes after model-bearing actions through the authenticated usage snapshot route.

- **OAuth-only** signup (Google + GitHub) — no passwords, no password-storage liability, raises per-account scripted-abuse cost; the pressure point of a one-shot grant is account farming, and this is its brake.
- **Structural bounds stay** — the quota bounds total spend; these bound burst, storage, and abuse: a per-account **skill-count cap** (five, **built or imported** — import costs no model tokens and is the acquisition wedge, [#66](https://github.com/CrowBe/agentbranch/issues/66); held skills cost nothing), a per-user, per-capability **request rate limit** (incl. import fetches, [#73](https://github.com/CrowBe/agentbranch/issues/73)), the gateway byte budget, and request-shape limits.
- **Analysis capabilities are free forever** — the lint report, hero, visualise fallback, and export (copy + the standard skill folder `.zip`) spend no tokens and never touch the quota.
- Aggregate protection: provider-side cap + the graceful-degradation catch (see [§4](#4-locked-decisions)).

## 9. Deferred features & their seams

Designed-for, not built. Each reuses an existing seam, so it's a renderer/config swap — not a new pipeline. The two principles worth stating explicitly:

- **The skill-analysis seam is the spine** — visualise, the Rendered hero, export, lint, and triggering evals all plug into it. New capabilities should ask "what input, artifact, and renderer is this?" before "what service is this?"
- **Cross-runtime validation is an evaluation capability on the seam** — a provider swap inside one engine, not a parallel pipeline.

**Build order after the v1 slice** — with the cross-runtime validation engine in place, the differentiators continue with the **draft/promote UI** ([#129](https://github.com/CrowBe/agentbranch/issues/129)) riding the already-landed branch substrate, then the **skill tap's validation stack and publication substrate** ([#133](https://github.com/CrowBe/agentbranch/issues/133) — shipped; the outward-facing distribution half remains). Primitives phases 4–6 ([#126](https://github.com/CrowBe/agentbranch/issues/126)) wait for a demand signal. The **harness improvement loop** ([#118](https://github.com/CrowBe/agentbranch/issues/118)) is built end-to-end (see [*Harness improvement loop*](#harness-improvement-loop-admin) below); its recommendations sharpen as the corpus of real user runs grows.

**Deferred capabilities:**

- **Branching iteration** *(built — the one shipped item in this list; kept here because [§9.3](#93-branching-iteration--draft-main-version-promote) is its design home)* — safe-space iteration: a user edits and evaluates a **draft** without disturbing the version they trust, then **sets the draft as the main version** when satisfied. Several drafts can be open at once, so an un-promoted draft from a prior session is never stranded. Borrows git's *shape* without forcing git vocabulary onto the SMB-facing UI. Both halves have landed: the **data substrate** ([#128](https://github.com/CrowBe/agentbranch/issues/128)) — branch/promote on `SkillRepository` + the daily retention job — and the **evaluate-on-draft + promote surface** ([#129](https://github.com/CrowBe/agentbranch/issues/129)) — draft edits build on the branch, evaluation records pin the draft head, and the hero makes the draft-vs-main state legible with **Set as main version** / **Discard draft** controls. Full design in [§9.3](#93-branching-iteration--draft-main-version-promote).
- **Cross-runtime validation** *(built engine; richer surface later)* — the existing triggering battery run against selected runtime targets through the model gateway; returns per-target results and an Insight, with unconfigured targets marked honestly.
- **Interactive visualise canvas** ("Claude design"-style point-and-annotate) — IR→React Flow renderer; **point-and-annotate falls out for free**: node → its source-span → inject that span as precise context into the next chat turn.
- **Richer evals** — workflow evals (run the full loop, LLM-as-judge against rubrics) → regression evals (pin a scenario set, track score across edits = the retention hook).
- **Baseline skill corpus** ([#149](https://github.com/CrowBe/agentbranch/issues/149)) *(built)* — a repo-tracked set of 20 curated skills for common moderately-technical and non-technical workflows (email triage, calendar planning, meeting notes, and the like), with source prompts, prompt batteries, provenance, and content hashes. It *is* the frozen regression benchmark's set (below), feeds the triggering eval's distractor library, and is ready to seed Templates.
- **Insight agent** — today each evaluation produces its **Insight** (the plain-language interpretation rendered as Insights) from one bounded `generate` call. The richer future is a tool-using agent (`runAgent` + tools) that *investigates* a result — re-runs cases, inspects the skill — before explaining it. Cross-cutting: it would explain *any* evaluation kind, reusing the seam, and replaces the `generate` call without changing the `Insight` shape or the renderer.
- **Skill tap** (publish + Skill library) *(built end-to-end — the public tap repo lives at [CrowBe/agentbranch-tap](https://github.com/CrowBe/agentbranch-tap); kept here because [§9.1](#91-skill-tap--open-distribution-with-safety-badges) is its design home)* — a public installation source other tools add once and install from. Publishing is open; validation is offered as an opt-in **safety badge**, and unvalidated published versions are labelled potentially unsafe ([#133](https://github.com/CrowBe/agentbranch/issues/133)).
- **Own-portfolio collision detection** — once accounts hold many skills, eval a new skill against the user's existing ones for trigger overlap.
- **Paid top-ups / PAYG** — Stripe Billing Meters on the existing token + cost meter (the free quota is step one of this: the priced spend stream already exists); transparent margin. Pairs with **separate Anthropic keys/workspaces for paid users** the moment they exist, so free-quota exhaustion can't block paying customers.
- **Realtime voice interaction** — the interaction surface evolves from the slim typed drawer (v1) to a floating **voice-forward control** (mic primary, typing fallback, no transcript column). Preview-primary was chosen so this is an *evolution, not a rework*; v1 UI choices must not block it. Slim drawer will stay as rendered transcript even in v2.

### Harness improvement loop (admin)

Built ([#118](https://github.com/CrowBe/agentbranch/issues/118)). The harness improvement loop is the admin-only feedback system that reads stored evaluation records across sessions and recommends changes to the validation harness itself. It is the first use of the skill-analysis seam where `Input` is not a single `Skill`: input is a **cohort of evaluation records** (triggering eval + test run records, pinned to skill versions and harness versions), artifact is a **harness-recommendation report**, and renderer is an **admin report surface** (the `harness-recommendation` module; `/api/admin/harness-report`). Tier 1 is static correlation, so the capability is an **analysis capability** — zero tokens, runs offline; a future model-written tier would be an evaluation capability (`defineEvaluation`) spending with a `platform` accounting tag.

The recommendation targets are the versioned pieces of the harness, not arbitrary product text: the build-loop system prompt, lint rules, test-run data-generation generator, triggering-eval prompt-battery generator, and distractor library. For the two generators, the thing attributed and improved is the **generator prompt + schema**, not any one generated scenario or prompt battery output.

Attribution needs two axes: **skill version × harness version**. The skill version says what artifact was evaluated; the harness version says which build-loop prompt, lint rules, generator prompts, schemas, and distractors produced the outcome. Without both pins, a later report cannot tell whether a failure came from the skill, the harness, or a changed generator. The `harness_versions` table is the second axis; stored `eval_runs` and `test_runs` carry nullable pins to both so old records survive retention.

The one aggregate persistence surface is the **admin aggregate read** on the eval/test-run repositories (`listForAnalysis`, Prisma + memory). User-scoped reads remain the default and ownership stays enforced in query for ordinary product surfaces; the aggregate read is exposed only behind `isAdmin` and the same allowlist posture as the model console. Its read model is outcomes and features by construction — pass/fail, case expectations and rationales, fired lint rules (joined from the skill version's stored lint summary), timestamps, skill and harness ids — user identity, prompt text, and `SKILL.md` content never cross into it. Reading content would require an explicit consent boundary because cross-user content is sensitive even when the goal is product improvement; no such read exists.

The first recommendation tier is deliberately auditable: correlate the lint rules a skill version fired with its triggering-eval outcomes and propose lint-rule changes — reweight a rule whose presence tracks failure, review one that fires mostly on passing skills, mine lint-clean failures for a new rule — each traceable to its evidence run ids. A human applies any change as an ordinary reviewed diff; the report never mutates anything. The **regression benchmark** is the guardrail: the frozen set (the baseline skill corpus + its curated batteries, pinned by a set hash) is scored through the triggering eval's competitive selection and recorded per harness version (`benchmark_runs`, `/api/admin/benchmark`), so vN+1 must beat vN on identical ground before a recommendation counts as an improvement. This is the cross-session cousin of **eval feedback**: eval feedback helps Claude revise one skill in the build loop; the harness improvement loop helps admins improve the evaluators and prompts that produce those results.

### 9.1 Skill tap — open distribution with safety badges

Built end-to-end ([#133](https://github.com/CrowBe/agentbranch/issues/133)). In-app: the full validation stack the badge rides on (lint policy rules, adversarial negative battery, `safetyReviewCapability`), the publication domain (`publications`, [§6](#6-data-model-sketch)), the publish route, the pure distribution read models and file renderers — the tap marketplace manifest, the standard skill folders for the tap repo, the tap repository file-set route (`/api/tap-repository`), and the Skill-library feed (`/api/skill-library`, Templates as a view over the reviewed tier) — and the workspace Templates / Skill library surface that consumes the feed with search, direct slug lookup, badge/flag, trust tier, content hash, and footprint copy at the install boundary. Public: the tap repo is [CrowBe/agentbranch-tap](https://github.com/CrowBe/agentbranch-tap), whose **publish-sync workflow** turns the `/api/tap-repository` snapshot into an auto-merged bot PR (applied via the source-to-tap sync script) and whose **advisory-checks workflow** runs the lint policy rules in the open on every PR; a publish in the app fires a `repository_dispatch` at it (the guarded tap-sync trigger — `TAP_REPOSITORY`/`TAP_SYNC_TOKEN`), with the workflow's scheduled sweep reconciling anything a missed dispatch leaves behind. "Tap" (an installation source you add once, then install from by name) is the internal term; user-facing copy says **Publish** / **Skill library**. A tap is *distribution*, not deploy: a published skill gets a public address, nothing runs.

**Publishing is open — validation is offered, never enforced.** No moderation step gates a publish. Instead, honesty does the work: a published version that carries a passing **safety rating** (below) pinned to its exact content hash renders a **safety badge**; a published version without one is labelled **potentially unsafe — not validated**. A changed version means a changed hash, so a revision is simply unbadged until the owner re-runs the analysis — the badge can never drift from the content it reviewed. Everything the old gate would have run stays available as the badge's substance and as authoring-time guidance (lint policy findings, adversarial battery, test run); none of it blocks distribution.

**Threat model.** Skills carry no code, but that bounds *our* platform, not the consumer's runtime — a published `SKILL.md` installs into agents holding shells, browsers, and credentials. Model a skill as **a program whose interpreter is an LLM**. Classes: (1) **direct malicious instructions** (the skill is the payload, the host agent the executor); (2) **agent hijacking** — overriding the host's guardrails; (3) **trigger hijacking** — a deceptive `description` fires the skill where it has no business, the tap-specific risk since the standard selects on descriptions; (4) **latent payloads** — benign on read, malicious on a date/keyword/fetched URL; (5) **reference-file smuggling** — every gate layer reads the whole folder, not just `SKILL.md`; (6) **typosquatting / impersonation**. Hard limit, accepted: class 4 isn't solvable by review — anything checked once is defeated by indirection. This threat model is why the honesty labelling exists: it is the live rationale for the badge/flag split and the install-boundary copy. *Content policy* remains its one static countermeasure — instructing the agent to fetch-and-follow remote instructions is a hard lint failure — but it is authoring guidance and badge substance, not a publish gate: a version tripping it can be published (flagged), it just can't be badged.

**The badge is the seam** — agent.branch is a validation tool that also distributes, not a file host bolting a rating on. The badge's substance reuses the seam: **skill lint** carries the static policy rules (shell-exec patterns, credential paths, fetch-and-follow indirection, obfuscation), reading the whole folder when reference-file contents are provided; the **triggering eval**'s negative battery includes fixed adversarial prompts, making it the trigger-hijack scanner; the **test run** catches behaviour the text hides (exfiltration-shaped tool calls show in the transcript); and the **`safetyReviewCapability`** — an LLM-judge evaluation capability, caller-tagged — reads the full folder and scores injection/exfiltration/deception. The judge is **structurally untrusted**: it reads the skill strictly as data, and its verdict is a claim about the artifact, never a guarantee about a runtime.

**Safety ratings are opt-in — manual, never automatic — and earn the badge.** The `safetyReviewCapability` surfaces as a **safety rating** the user runs on demand. A rating pins the version it reviewed; versions are append-only, so a rating stays true forever and a new revision is simply *unrated* — the scan is offered any time for an unrated version (the hero's **Safety** chip) and as an **optional step of "Set as main version"** (skippable, and a worrying verdict only asks — it never blocks promote or publish). A published version whose head carries a passing rating renders the **safety badge**; an unrated or non-passing published version carries the **potentially unsafe — not validated** label instead. Because the user asks for it, the run is `account`-tagged against the `safety-review` capability — nobody spends quota on a rating they didn't request; the capability stays caller-tagged, so any future platform-initiated review would spend `platform`. A rated version never re-spends: reopening the rating and switching Insights/breakdown render the stored record. Ratings persist as `safety_ratings` ([§6](#6-data-model-sketch)), pinned to skill version + harness version like every evaluation record.

**Tiered visibility — open existence, humans for amplification.** Effort scales with what we promote, not with volume:

| Tier | Bar | Reach |
|---|---|---|
| **Private** | — | Default; owner only (status quo). |
| **Published** | None — publishing is open | In the tap, link-reachable, installable. Carries the **safety badge** (passing rating pinned to this hash) or the **potentially unsafe — not validated** label. |
| **Reviewed** | Published **+ human review** | Surfaced: search, Templates, featured. What we vouch for — an editorial pick, not a moderation verdict. |

Supporting policy: publishers are attributable (OAuth-only identity, already true); publish pins an append-only skill version + **content hash** (a badge provably describes the thing served; new version = unbadged until re-analysed); publish attempts are **rate-limited** via the existing per-capability windows; slugs live under the publisher's handle (`owner/skill-name`), removing most squatting value.

**The tap is a public git repo — everything visible, nothing gated.** The repo carries a `.claude-plugin/marketplace.json` (add once in Claude Code, install by name) and **is the publication mechanism**, Homebrew-style: publish = bot PR adding the rendered standard skill folder pinned to a version + hash, auto-merged — no check blocks it. The **lint policy rules still run as open-source CI in the tap repo** and the hosted layers still post verdicts to the PR, but as **advisory annotations** feeding the badge/flag metadata, not merge conditions. The **reviewed** tier is a curated index changed only by human-approved PRs, so vouching is itself a public, attributable diff; **takedown = revert at HEAD** (installs read HEAD, so a revert ends installability immediately — the abuse backstop that replaces up-front gating), with reports via the tap repo's "Report a skill" issue form and the profile page's report link (slug + content hash prefilled, so a report always names the exact bytes it is about). Accepted residuals, named: git history persists (revert immediately, purge via host support for confirmed malware — tolerable for instruction text where it wouldn't be for binaries); static rules are public (attackers can test offline; transparency buys more trust than the obscurity protects); and **unbadged malicious content can be live until reverted** — the model accepts detection-and-takedown plus labelling in place of prevention, which is why the flag copy must be blunt.

**Consumer-side honesty.** We can't make another runtime safe and the standard has no permissions manifest. At the install boundary we surface what analysis can derive: the skill's tool/behaviour footprint, trust tier, the safety badge or the potentially-unsafe label, rating results when they exist, content hash. Presentation, never a guarantee — copy says so.

**Public skill profile pages.** Every visible publication gets a public page at `/skills/<owner>/<name>` — the discovery/SEO surface of the tap. The page is a pure render over the pinned version and its safety state (profile read model + the hero capability's `rendered` view), served force-dynamic so a takedown revert or a new rating is visible immediately; it can never describe content it doesn't match, for the same reason the badge can't drift. Category and tags from the pinned frontmatter drive the Skill library's `?category=` / `?tag=` filters and tag-aware search; the same metadata is how "top skill in its class" curation gets its classes — curation itself stays the human-approved reviewed tier, never a score leaderboard.

**Build order:** the validation stack (static lint policy, adversarial negative battery, `safetyReviewCapability` + opt-in safety rating), the publication domain (pinned version, slug, tier, hash, rate limit), the distribution read models (tap marketplace manifest, tap repository file set, Skill-library feed with Templates as a view over the reviewed tier), the publish route, the workspace Templates / Skill library surface, and the outward-facing half — the public tap repo with its advisory CI and publish-time bot wiring ([#173](https://github.com/CrowBe/agentbranch/issues/173)) — all run today.

### 9.2 Equipment primitives & composition (broadening beyond Skill)

Tracked in [#126](https://github.com/CrowBe/agentbranch/issues/126) (paired with branching iteration, [§9.3](#93-branching-iteration--draft-main-version-promote)). **The first two primitives and the first composition are built** ([#130](https://github.com/CrowBe/agentbranch/issues/130)–[#132](https://github.com/CrowBe/agentbranch/issues/132)); later primitives stay designed-not-built. The seam is built so `Skill` is the *first* concrete input, not the only possible one: `Analyzer<Input, A>` / `Evaluator<Input, A>` take a generic input ([§2](#2-glossary--the-domain-language), [§3.1](#31-the-skill-analysis-seam-the-spine)). Broadening the product means adding more **equipment primitives** — declarative artifacts that change what an agent can do, know, decide, or safely access — each with its own source model, analyzer/lint, renderer, and optionally an evaluator, all on the existing seam (no new pipeline).

The broadening earns its keep only when a **second** primitive ships and **composition** becomes possible: cross-primitive evaluation questions the single-primitive product can't ask —

- Does this Skill call this Tool correctly?
- Does the Tool's output match the Response schema?
- Does the Skill stay inside the declared Policy?

Composition is **additive on the seam** — an evaluator gets richer input context, it does not become a new pipeline.

**Primitive order (each earns the next):**

1. **Response schemas** *(built, [#130](https://github.com/CrowBe/agentbranch/issues/130); chat-buildable, [#151](https://github.com/CrowBe/agentbranch/issues/151))* — structured output definitions (JSON Schema documents). The `response-schema` module carries the lossless source model and a pure, zero-token lint (structural validity, required fields, smell rules) rendering through the seam renderers, plus the offline schema-subset validator and example-value generator the compositions below use. Authoring rides the build loop's shape: one frozen cacheable system prompt (interview-first, anchored on one real filled-in example of the output) with a `write_response_schema`/`edit_response_schema` tool pair through the model gateway, surfaced in the workspace's Equipment mode; the finished document is quality-checked and session-kept exactly like a pasted one. Each later primitive gets the same treatment — its own frozen prompt + tool pair — as it becomes chat-buildable.
2. **Tool contracts** *(built, [#131](https://github.com/CrowBe/agentbranch/issues/131); backend authoring path, [#175](https://github.com/CrowBe/agentbranch/issues/175))* — typed input/output plus descriptions, examples, failure modes, safety notes. The `tool-contract` module mirrors the same shape (lossless source model + pure lint, inline schemas run the shared schema shape rules); its I/O references response schemas via `$ref` by title. Authoring rides the same build-loop shape as response schemas: one frozen cacheable prompt anchored on the tool's job, example input/output, failure modes, and confirmation boundary, with a `write_tool_contract`/`edit_tool_contract` pair through the model gateway and lint feedback after writes. The first relational evaluation is also built ([#132](https://github.com/CrowBe/agentbranch/issues/132)): **Test run** takes a **bundle**, the mock-tool registry is driven by the contracts (mock output conforms to the output schema), and each observed call is validated — arguments against the input schema, handled output against the output schema — surfacing in Insights, the breakdown, and eval feedback.
3. **Policies / guardrails** — declarative constraints on allowed actions, confirmations, and data/network/file/output access. Lint catches obvious violations; evaluation tests whether the constraint holds under pressure.
4. **Agent profiles** (later) — durable identity + defaults (system prompt, role, tone, model preference, default equipment). Analysis is easy; the evaluation half needs rubric / LLM-judge support.
5. **Knowledge packs** (later) — bounded reference knowledge, lightweight RAG-adjacent, no vector DB. Meaningful only when evaluated as part of a profile + skill + tool bundle.

Supporting work on the way: **user-authored test materials** (let users seed Triggering eval / Test run with their own positive/negative prompts and scenarios — extends existing evaluators before becoming a top-level primitive) and **regression comparison** across evaluation records ([§6](#6-data-model-sketch); Richer evals above) so a revision's improvement or regression is visible.

**Smallest useful composition** *(built — the bundle test run above)* — the bar for being more than a single-primitive tool: *here is a Skill, a Tool contract, and a Response schema; does the Skill call the Tool correctly and produce valid output?* Broader than single-primitive validation without dragging in container runtime, vector DBs, or subagent routing — the **non-goals for the first expansion**.

### 9.3 Branching iteration — draft, main version, promote

**This is built** — data substrate ([#128](https://github.com/CrowBe/agentbranch/issues/128)) and user-facing surface ([#129](https://github.com/CrowBe/agentbranch/issues/129)). The schema, the branch/promote repository methods, and the daily retention job ([§6](#6-data-model-sketch)) live behind the `SkillRepository` + `SkillRetentionRepository` ports; the surface wires them into build + evaluate: a draft build appends to the branch head (`saveToBranch`, main pointer untouched), evaluation records pin the draft head (a shared version resolver over `branchId`, [§3.1](#31-the-skill-analysis-seam-the-spine)), and the hero carries the draft-vs-main legibility banner with the **Start a draft** / **Set as main version** / **Discard draft** controls plus a resume affordance for open drafts. Versioning is **draft / main / promote** so a user can iterate and evaluate without disturbing the version they trust.

**The model.** A skill has one **main version** — the blessed pin everything defaults to (export, install, the hero's first paint). Iteration happens on a **draft**: a lineage of revisions that accumulates *without moving the main pointer*. The user evaluates the draft (lint, test run, triggering eval all run against the draft's head, not main), and when satisfied **sets it as the main version** — promote moves the pin to the draft's head. A skill may carry **several open drafts at once**: a new build session forks a fresh draft off main or resumes an open one, so an un-promoted draft from a prior session is never stranded and never silently becomes main. This is what made multiple drafts the call over a single draft slot — a one-draft model forces a stale/resume/discard decision the moment a second session starts; coexisting drafts remove that state machine.

**Git shape, stated precisely.** Internally, *main is just a branch* — every version belongs to a branch, one branch is flagged the blessed/main lineage via an explicit pointer, and promote re-points it. Versions chain by **parent pointer** (a DAG), so two drafts advancing independently never collide — this is why the linear monotonic `revision` demotes to a per-draft display ordinal ([§6](#6-data-model-sketch)). **Promote is replace-not-merge:** setting a draft as main repoints the pin to that draft's head; it is *last-promote-wins* with no three-way merge. Instruction-only skills carry no merge-conflict semantics worth solving, so a draft based on a now-superseded main stays a valid draft the user can re-promote (overwriting) or discard. **No merge, no rebase** — the accepted simplification that keeps the SMB model a single verb.

**Draft ≠ session.** The draft is the durable artifact; a build session is ephemeral. A session records which draft it last touched (for resume), but a draft outlives its session — there is no session entity ([§6](#6-data-model-sketch)), so resume is a pointer, not a lifecycle.

**Copy contract — git shape, not git vocabulary.** User-facing nouns and verbs: a branch is a **draft**; the blessed version is **the main version**; promote is the button **"Set as main version"** ("main" is *semantic* — the main one — not git's branch name). The internal/code terms `branch` and `promote` never reach user copy, the same internal-vs-surface split as mock-tool registry / test run and tap / Publish ([§9.1](#91-skill-tap--open-distribution-with-safety-badges)). **"Publish" is reserved for the tap** — where a skill genuinely becomes visible to others — and must not be reused for promote, which changes nothing's visibility.

**Retention is a daily tidy, not an inline cap.** Pruning is off the write path (no prune inside a save/promote transaction): it runs as a **fixed-cadence cleanup job** — the `SkillRetentionRepository` port, driven by a **Vercel Cron** trigger (`/api/cron/retention`, locked behind `CRON_SECRET`) — which the UI can state plainly ("older drafts are tidied up daily; we keep your last N per draft and your main version"). The job enforces two axes, neither during a live session: **per-draft history depth** (latest-N-per-draft) and a structural **open-drafts-per-skill cap** (one tunable constant, the same bounded-by-construction shape as the skill-count cap, [§8](#8-free-quota)). It **never prunes the main lineage or any open draft's tip** — only interior history beyond N. Moving retention off the write path also dissolves the "don't prune an in-flight draft out from under the user" hazard *by construction*: nothing is pruned while a session is live. Schema delta + the retention port live in [§6](#6-data-model-sketch).

**On the seam.** Branching is **upstream of the skill-analysis seam**, not a capability on it: it changes *which `SkillSource`* feeds an analyzer/evaluator, never the seam's `artifact → render` shape. No new `ArtifactKind`, no renderer change — every capability (hero, lint, visualise, export, test run, triggering eval) runs against a draft head exactly as it runs against main today. The substrate is therefore a **port + adapter** change (MODULE_DESIGN §6 rule 2) on the existing `SkillRepository` plus one new retention port — not a new pipeline — implemented in both the Prisma and memory adapters and tested as one contract.
